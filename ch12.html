基礎講座 第12章

クラウド・データセンター・仮想化 — 午後IIの主戦場を歩く

午後IIの舞台は、この10年で「社内LANの設計」から「クラウドとデータセンターを含めた全体設計」へ移った。この章では、クラウド接続方式の選定、スパイン&リーフとVXLAN/EVPN、SDN/NFV、IoTまで、2023〜2025年度の午後IIで実際に問われたテーマ群を一気に押さえる。

学習目標:
  • 要件からクラウド接続方式(インターネットVPN/閉域接続/専用線)を選定し、理由を記述できる
  • スパイン&リーフ構成の利点を、従来の3階層構成と対比して説明できる
  • VXLANのカプセル化構造と、VNI・VTEP・アンダーレイ/オーバーレイの関係を説明できる
  • EVPNがBGPを使って何を解決するかを、フラッディング学習と対比して説明できる
  • MQTTやLPWAなど、IoTシステム特有のネットワーク技術の特徴を説明できる

想定学習時間: 3.5〜4.5時間(例題・章末テスト込み)

12-1企業ネットワークの重心変化 — 守るべきものはもう社内にない

かつての企業ネットワークは単純な同心円だった。中心にオンプレミスのサーバ室(またはデータセンター)があり、各拠点はWANで本社につながり、インターネットは「たまに使う外の世界」。設計の主眼は社内のサーバへいかに速く安全に到達させるかであり、インターネットとの境界はFWで固く閉じておけばよかった。

この前提が崩れた。業務システムはSaaSへ、自社サーバはIaaS上の仮想マシンへと移り、社員のトラフィックの多くが社内ではなくクラウドに向かうようになった。すると設計の中心課題も変わる。「本社までのWANをどう作るか」ではなく、「クラウドとどう接続するか」「クラウド宛のトラフィックをどこで検査し、どこからインターネットに出すか」が主戦場になったのだ。

試験もこの変化を忠実に追いかけている。事実として、午後IIでは2023年度に「マルチクラウド利用による可用性向上」、2024年度に「データセンターNWの検討(VXLAN/EVPN/VTEP)」、2025年度に「社内ネットワークのIPv6対応」と「IoTシステムの設計」が出題された。本章はこの3年分の午後II頻出領域を正面から扱う。第11章のIPv6と合わせれば、近年の午後IIの出題領域はほぼ覆える。

合格の視点クラウド系の問題は「新技術の知識問題」に見えて、実際は要件と制約から構成を選ぶ判断問題だ。個々の用語(VXLAN、SASE、MQTT)は道具にすぎない。各節で「何の問題を解決する道具か」を必ずセットで覚えること。それがそのまま記述解答の骨格になる。

12-2クラウド接続方式の選定 — インターネットVPNか、閉域か

オンプレミスの拠点とクラウド(IaaS)を結ぶ方法は、大きく2系統ある。インターネットVPNと閉域接続だ。午後問題では「この要件ならどちらを選ぶか、理由とともに述べよ」という形で繰り返し問われるので、それぞれの特性を対比で頭に入れる。

  • インターネットVPN: インターネット上にIPsecなどで暗号化トンネルを張る方式。安価で開通が速い(既存のインターネット回線があれば設定だけで始められる)。ただし経路は共用のインターネットなので、帯域や遅延はベストエフォート——混雑すれば遅くなり、保証はない。
  • 閉域接続: 通信事業者の閉域網(IP-VPNや広域イーサネット)からクラウド事業者の専用接続サービスへ直結する方式。帯域・品質(遅延・パケット損失)のSLAが提供され、トラフィックがインターネットを一切通らないため、盗聴やDDoSの影響を受けにくい。代償はコストと開通までのリードタイムだ。
  • 専用線: 拠点とデータセンター等を1対1で物理的に結ぶ最上位の選択肢。品質・機密性は最高だが最も高価で、現在は「閉域網へのアクセス回線」として使われることが多い。
観点インターネットVPN閉域接続・専用線
帯域・遅延の保証なし(ベストエフォート)あり(SLA)
機密性暗号化で担保(経路はインターネット)経路自体が閉域。インターネット非経由
コスト安価高価
開通までの期間短い長い(物理工事や事業者手続)
典型用途小規模拠点、バックアップ回線基幹システム、大容量・安定要件

実戦での定番構成は組合せだ。基幹システム向けの閉域接続を主回線とし、障害時のバックアップとして安価なインターネットVPNを併用する。「主が閉域・従がVPN」という非対称な冗長化は、コストと可用性のバランス解として午後の模範解答に頻出する。逆に、応答時間の安定が必須の基幹系をインターネットVPNだけで結ぶ提案は、要件を満たせないとして修正対象になる。

試験ではこう出る「D社が基幹システムのクラウド移行に当たり閉域接続を選定した理由を述べよ」→「帯域と遅延の品質保証があり、インターネットを経由しないため通信内容の漏えいやインターネット障害・攻撃の影響を受けにくいから」。理由は品質保証と閉域性の2軸で書くのが型。逆方向(コスト重視でVPNを選ぶ理由)も書けるように。
例題 12-1ある企業が販売管理システムをIaaSへ移行する。要件は「応答時間が安定していること」「通信がインターネットを経由しないこと」である。選定すべき接続方式を挙げ、その方式が2つの要件をそれぞれどう満たすかを70字程度で述べよ。
解答と解説

解答例:「閉域接続を選定する。帯域・遅延がSLAで保証されるため応答時間が安定し、通信事業者の閉域網からクラウドへ直結するためインターネットを経由しない。(71字)」

要件が2つ提示されたら、解答でも要件ごとに対応する特性を1対1で述べる。「安全だから」「高品質だから」のような要件に紐づかない一般論は得点にならない。

12-3ハイブリッドクラウドとマルチクラウド — 2023年度午後IIの世界

用語を先に整理する。オンプレミスとクラウドを組み合わせて使うのがハイブリッドクラウド、複数のクラウド事業者を併用するのがマルチクラウドだ。マルチクラウドの主目的として試験で問われるのは可用性の向上——単一のクラウド事業者で大規模障害が起きてもサービスを止めないための構成であり、まさに2023年度午後IIが「マルチクラウド利用による可用性向上」というテーマだった。副次的な目的として、特定事業者への依存(ロックイン)の回避や、サービスごとの適材適所もある。

ネットワーク設計者の視点では、マルチクラウドは「接続をどう束ねるか」の問題になる。各拠点から各クラウドへ個別に接続を張るとフルメッシュ状に膨れ上がるため、閉域網をハブにしたハブ&スポーク型が定石だ。

        本社          拠点A         拠点B
          │             │             │
          └──────┬──────┴─────────────┘
                 │
        〔閉域網(IP-VPN/広域イーサ)〕 ←ハブ
           │           │           │
       [接続点]     [接続点]     [接続点]
           │           │           │
       クラウドX    クラウドY    オンプレDC
       (本番系)    (待機系)   (既存システム)

図12-1: 閉域網をハブとしたマルチクラウド構成。拠点は閉域網に入りさえすれば、どのクラウドにも到達できる。

可用性設計の核心は切替の仕組みだ。本番系のクラウドXが障害になったとき、利用者を待機系のクラウドYへどう誘導するか。代表的な答えがDNSによる切替である。サービスのFQDNが返すIPアドレスを、X宛からY宛へ書き換える(あるいはヘルスチェック連動で自動的に切り替えるGSLB——広域負荷分散——を使う。第7章)。このときの設計論点がTTLだ。DNS応答のTTLが長いと、切替後もキャッシュされた古いアドレスが使われ続け、切替が浸透するまでの時間が延びる。「事前にTTLを短くしておく」は午後の記述で書ける定番の運用だ。

もうひとつの論点はデータと状態。ネットワークが切り替わっても、待機系にデータが同期されていなければサービスは再開できない。DB複製の方式や、切替時に失われるセッションの扱いは、ネットワークの設問と絡めて出題される。答案でも「切替後に業務を継続できる根拠」まで意識して書けると強い。

合格の視点マルチクラウド問題は、第7章で学んだ冗長化の考え方(アクティブ/スタンバイ、ヘルスチェック、フェールバック)を「装置」から「クラウド事業者」へスケールアップしただけ、と捉えると見通しがよい。新しい概念はDNS切替のTTL管理と閉域ハブくらいで、思考の骨格は同じだ。

12-4SaaS時代のトラフィック設計 — プロキシ集約の限界とローカルブレイクアウト

従来の企業では、全拠点のインターネット向け通信をWAN経由で本社(またはDC)のプロキシ・FWに集約し、そこで一括して検査してからインターネットへ出す構成が主流だった。管理点が1箇所で済み、セキュリティポリシーを集中適用できる合理的な設計だ。

ところがSaaS利用が増えると、この構成は自壊し始める。拠点→本社のWAN回線と本社のプロキシに、本来インターネットへ直行すればよいSaaSトラフィックが全部乗ってくるからだ。WAN帯域の逼迫、プロキシの処理能力不足、そして拠点からSaaSまでの遠回りによる遅延——「Web会議が本社経由で音切れする」という象徴的な事態が起きる。

解決策がローカルブレイクアウトだ。信頼できる特定のSaaS宛の通信だけを、本社を経由させず拠点から直接インターネットへ出す。宛先のFQDNやIPアドレス帯でSaaS通信を識別し、拠点のルータで振り分ける。ただし、直接出すぶん本社での検査を受けなくなるので、セキュリティをどう担保するかという次の問題が生まれる。

その答えが検査機能自体のクラウド化、すなわちSWG(セキュアWebゲートウェイ)だ。各拠点・各端末からの通信をクラウド上のSWGに通し、URLフィルタやマルウェア検査を行ってからインターネットに出す。プロキシが「本社の装置」から「クラウドのサービス」になったと考えればよい。さらに、SD-WANなどのネットワーク機能とSWG系のセキュリティ機能を一体のクラウドサービスとして提供する考え方がSASEである。SWG・SASEのセキュリティ面の詳細は第8章で扱っているので、本章では「トラフィック設計の帰結としてなぜ必要になるか」という文脈を押さえてほしい。実際、2025年度午後IではSWGの導入が1問として出題されている。

試験ではこう出る「全拠点の通信を本社プロキシ経由とした場合の問題点を2つ述べよ」→「WAN回線とプロキシにSaaS宛の通信が集中し帯域・処理能力が逼迫する」「拠点からSaaSまでの経路が遠回りになり遅延が増える」。続けて「ローカルブレイクアウト対象とする通信の選定基準」「直接通信のセキュリティをどう担保するか(SWG)」まで一連で問われるのが近年のパターン。

12-5データセンターNWの構造転換 — 3階層からスパイン&リーフへ

ここから舞台をデータセンター(DC)内部に移す。伝統的なDCネットワークはコア・ディストリビューション・アクセスの3階層構成だった。アクセススイッチがサーバを収容し、ディストリビューションが集約し、コアが全体を束ねる。この構成は、トラフィックの大半が「サーバ→コア→外部」という南北方向(North-South)に流れる時代には合理的だった。

だが、サーバ仮想化と分散システムの普及で状況が変わる。Webサーバ→APサーバ→DBサーバ、あるいは分散ストレージのノード間といったサーバ同士の通信=東西方向(East-West)がトラフィックの主役になったのだ。3階層でこれを運ぶと、別のアクセススイッチ配下のサーバ宛の通信がいちいち上位層まで往復し、経路長がまちまちで遅延が読めない。さらにL2の冗長経路はSTP(第2章)がブロックしてしまうため、せっかくのリンクの半分が遊ぶ。

この問題への答えがスパイン&リーフ(リーフ・スパイン)構成だ。サーバを収容するリーフスイッチ群と、リーフ同士を中継するスパインスイッチ群の2階層とし、すべてのリーフをすべてのスパインに接続する。

【従来: 3階層】             【スパイン&リーフ: 2階層】
     [コア]  [コア]           [スパイン1]   [スパイン2]
       │ ╲  ╱ │               │ │ ╲ ╲   ╱ ╱ │ │
       │  ╳   │               (全リーフ⇔全スパイン)
       │ ╱  ╲ │               │ │ ╱ ╱   ╲ ╲ │ │
 [ディストリ][ディストリ]    [リーフ1] [リーフ2] [リーフ3]
   │   │      │   │           │        │        │
 [アクセス]…[アクセス]…     サーバ群  サーバ群  サーバ群
   │            │
 サーバ群     サーバ群       どのサーバ間の通信も
 南北中心・STPでブロック     リーフ→スパイン→リーフで等距離

図12-2: 3階層とスパイン&リーフ。後者はどの2台のサーバ間も同一ホップ数で結ばれ、ECMPで全リンクを使える。

スパイン&リーフの利点は3つの言葉で整理できる。第一に等距離性——どのサーバ間の通信も必ず「リーフ→スパイン→リーフ」を通るため、ホップ数が均一で遅延が予測しやすい。第二に全リンク活用——リーフとスパイン間をL3(IPルーティング)で構成し、ECMP(等コストマルチパス)で複数経路に負荷分散する。STPのように冗長リンクを遊ばせない。第三にスケールアウト——帯域が足りなくなったらスパインを1台追加すれば全リーフ間の帯域が底上げされ、サーバ収容を増やしたければリーフを足せばよい。装置を大型機に置き換える(スケールアップ)のではなく、台数で伸ばせる。

ただし、リーフ〜スパイン間をL3で作るということは、異なるリーフ配下は別のIPセグメントになるということだ。ここで次節の「仮想化は同一L2セグメントを要求する」という話と衝突する。この矛盾を解くのが12-7節のVXLANである——という流れを頭に置いて読み進めてほしい。

例題 12-2データセンター内でEast-Westトラフィックが支配的なシステムに対し、3階層構成に代えてスパイン&リーフ構成を採用する利点を、経路の観点と帯域の観点から1つずつ、それぞれ40字程度で述べよ。
解答と解説

解答例(経路):「どのサーバ間の通信もリーフ→スパイン→リーフの同一ホップ数となり、遅延が均一で予測しやすい。(46字)」

解答例(帯域):「ECMPで複数スパイン経由の経路に負荷分散でき、スパインの追加で帯域を拡張できる。(41字)」

「速い」「効率的」という形容だけでは得点にならない。なぜ速いのか(等距離)、なぜ広がるのか(ECMP・スケールアウト)という構造上の根拠を書くこと。

12-6サーバ仮想化とネットワーク — ライブマイグレーションがL2延伸を要求する

現代のDCの主役は物理サーバではなく、ハイパバイザ上の仮想マシン(VM)だ。仮想化はネットワークに2つの新しい登場人物を連れてくる。

ひとつは仮想スイッチ(vSwitch)。ハイパバイザ内部で動くソフトウェアのL2スイッチで、VMの仮想NICを収容し、物理NIC経由で外部と接続する。重要な帰結として、同一ホスト上のVM同士の通信は物理ネットワークを一切通らない。物理スイッチのミラーポートで監視していてもこの通信は見えない——トラフィック可視性という設計論点が生まれる。VLAN(第2章)の付与も仮想スイッチが担い、物理NICへはタグVLANで送り出すのが一般的だ。

もうひとつがライブマイグレーション。稼働中のVMを、通信をほぼ止めずに別の物理ホストへ移動させる機能だ。ホストの保守や負荷の平準化に必須だが、ネットワークに強い制約を課す。移動の前後でVMのIPアドレスとMACアドレスは変わらない——IPが変わればTCPコネクションもセッションも切れてしまうからだ。IPを維持したまま別ホストで動き続けるには、移動元と移動先のホストが同一のL2セグメント(同一サブネット)に接続されている必要がある。移動後にVM側からGratuitous ARPなどを送ってスイッチのMAC学習を更新させ、トラフィックを新しい場所へ引き寄せる——これはL2の仕組みそのものだ。

すると要求はこうなる。「ラックをまたいでも、フロアをまたいでも、場合によってはDC間でも、同じL2セグメントを届かせたい」——いわゆるL2延伸だ。しかし素朴にVLANを広げると、ブロードキャストドメインが際限なく肥大し、ループやSTP再計算のリスクが全域に波及する。しかも前節で見たとおり、モダンなDCファブリックはL3で作られており、素のVLANはリーフを越えられない。「L3ファブリックの上で、離れた場所に同一のL2セグメントを提供する」——この一見矛盾した要求を解くのが、次節のVXLANである。

試験ではこう出る「ライブマイグレーションの移動元と移動先が同一セグメントに接続されている必要がある理由を述べよ」→「移動の前後でVMのIPアドレスを変えずに通信を継続するため」。仮想スイッチ側は「同一ホスト内のVM間通信が物理ネットワーク上で観測できない理由を述べよ」。どちらも仕組みを1文で言えれば得点できる。

12-7VXLAN詳解 — L3の海の上にL2の橋を架ける(2024年度午後IIテーマ)

VXLANが解決する問題は2つある。第一にVLAN数の限界。VLAN IDは12ビットで、使える番号は実質4094個。テナント(顧客・部門)ごとに多数のセグメントを切るマルチテナントDCでは足りない。第二に、前節で見たL3ファブリック上でのL2延伸だ。

VXLANはこの2つを一挙に解決する。発想は大胆で、元のL2フレームを丸ごとUDP/IPパケットの中に包んで(カプセル化して)運ぶ。セグメントの識別子はVNIという24ビットの番号で、224=16,777,216——約1677万個のセグメントを識別できる。VLANの4094個との桁違いの差は午前IIの定番だ。

│ヘッダ     │ヘッダ     │宛先ポート│ヘッダ  │(宛先MAC・送信元│
│          │(VTEP間の │4789    │VNI       │MAC・IPパケット)│
│          │IPアドレス)│        │(24ビット)│               │

             └──アンダーレイのルータが見るのはここだけ──┘

図12-3: VXLANのカプセル化構造。元のフレームはUDPペイロードとして運ばれ、途中のネットワークからは単なるIP/UDP通信に見える。

カプセル化と復元を行う出入口がVTEP(VXLANトンネルエンドポイント)だ。リーフスイッチや仮想スイッチがVTEPになる。VM-AからVM-Bへのフレームは、A側のVTEPでVXLANヘッダ+UDP+IPを被せられ、宛先VTEPのIPアドレスへ普通のIPパケットとしてルーティングされ、B側のVTEPで脱がされて元のフレームに戻る。

ここで必ず押さえたいのがアンダーレイとオーバーレイという2層の見方だ。

【オーバーレイ(論理)】
  VM-A ═════════ 同一L2セグメント(VNI 10100) ═════════ VM-B
         VTEP間に張られた論理的なトンネルの上のL2世界

【アンダーレイ(物理)】
  VM-A                                              VM-B
   │                                                 │
 [リーフ1=VTEP]──[スパイン群]──[リーフ2=VTEP]
  IP 10.0.0.1    (IPルーティング+ECMP)   IP 10.0.0.2
   └──カプセル化したUDP/IPパケットを転送するだけ──┘

図12-4: アンダーレイとオーバーレイ。物理網はVTEP間のIP到達性だけ提供し、論理的なL2セグメントはその上に浮かぶ。

アンダーレイは物理のIPネットワークで、仕事は「VTEP同士がIPで到達できること」だけ。スパイン&リーフのL3ファブリックがそのまま使え、ECMPの恩恵も受けられる(カプセル化がUDPなのは、送信元ポート番号をフローごとに変えることでECMPのハッシュ分散を効かせる狙いもある)。オーバーレイはその上に構築される論理ネットワークで、VMたちからは普通のL2セグメントに見える。かくして「L3で作った丈夫な物理網」と「仮想化が要求するL2の同一セグメント」が両立する。前節の矛盾はこう解けた。

残る課題がBUMトラフィック(Broadcast/Unknown unicast/Multicastの頭文字)だ。ブロードキャスト(ARPなど)や宛先未学習のフレームは「同一セグメントの全員」に届けなければならないが、オーバーレイの「全員」は複数のVTEPの向こうに散らばっている。素朴な解決は、同じVNIに参加する全VTEPへコピーを送りつける方式(イングレスレプリケーション)か、アンダーレイのマルチキャストを使う方式。いずれも無駄なフラッディングが発生しがちで、この非効率をコントロールプレーンで解決するのが次節のEVPNだ。

例題 12-3VXLANに関する次の記述の空欄を埋めよ。
「VXLANでは、元のL2フレームを〔 a 〕/IPでカプセル化して転送する。セグメントは〔 b 〕ビットの識別子〔 c 〕で区別され、約〔 d 〕個のセグメントを収容できる。カプセル化と復元は〔 e 〕と呼ばれる装置・機能が行う。」
解答と解説

a: UDP b: 24 c: VNI d: 1677万(16,777,216) e: VTEP

2024年度午後IIはまさにこの用語群(VXLAN/EVPN/VTEP)が題材だった。「VLANは12ビットで4094個、VNIは24ビットで約1677万個」という対比は数字ごと覚えてしまうこと。カプセル化のトランスポートがTCPではなくUDPである点もひっかけで問われる。

12-8EVPN — 「フラッディングで学習」から「BGPで広告」へ

VXLANはあくまでデータプレーン(運び方)の規格であり、「宛先MACアドレスがどのVTEPの向こうにいるか」をどう知るかは別問題だ。素のVXLANは従来のL2スイッチと同じフラッディング&学習方式に頼る。つまり、宛先が分からなければ全VTEPへ撒き(フラッディング)、返ってきたフレームの送信元から「このMACはあのVTEPの先」と学習する。小規模なら動くが、VTEPとVMが増えるほど無駄なBUMトラフィックが増え、学習が揮発するたびに再フラッディングが起きる。データセンター規模ではスケールしない。

そこでEVPNだ。発想を転換し、MACアドレスの所在を「流して学ぶ」のではなく「経路情報として広告する」。使うプロトコルはインターネットの経路交換で実績のあるBGP(MP-BGPの拡張)で、各VTEPは自分の配下にいるVMのMACアドレス(およびIPアドレス)を、BGPの経路としてほかのVTEPへ広告する。つまりEVPNはVXLANのコントロールプレーンを担う。

観点フラッディング&学習(素のVXLAN)EVPN(BGPで広告)
MAC所在の学習実トラフィックを撒いて、返りから学ぶVTEPがBGP経路として事前に広告
未知宛先のフラッディング頻発する所在が既知なので大幅に削減
ARP要求への応答全VTEPへ転送VTEPが学習済み情報で代理応答(ARP抑制)
VTEPの発見設定などで個別に対応BGP経由で自動発見
スケール小規模向け大規模DC・DC間接続向け

EVPNの利点は「フラッディング削減」に集約されるが、その波及効果も試験で問われる。学習済みのMAC/IP対応を使ってVTEPがARP要求に代理応答すれば、ARPブロードキャストがオーバーレイを渡らずに済む(ARP抑制)。VMがライブマイグレーションで別のリーフへ移動すれば、移動先のVTEPが新しい経路を広告して全体が即座に追従する。複数のリーフにサーバを二重接続するマルチホーミングの制御もBGPの仕組みに乗せられる。「データプレーンはVXLAN、コントロールプレーンはEVPN(BGP)」という役割分担の一文は、そのまま記述解答に使える型として覚えておこう。

合格の視点BGPを「インターネットの経路交換」としか知らないと、EVPNで急に出てきて面食らう。実体は「信頼性の高い経路広告の仕組みを、IPプレフィックスではなくMAC/IPの所在情報に流用した」だけだ。2024年度午後IIでは、この構成のDCネットワーク設計が2時間まるごと問われた。仕組みの絵(図12-4)を白紙に再現できるようにしておくこと。

12-9SDNとNFV — 「制御の分離」と「機能の仮想化」を混同しない

ネットワーク機器は伝統的に、経路を判断する頭脳(コントロールプレーン)とパケットを転送する手足(データプレーン)を1台の中に同居させてきた。機器が増えるほど、設定は1台ずつ、全体像は誰も持っていない、という運用になりがちだ。

SDN(Software-Defined Networking)はこの構造を壊す。コントロールプレーンを機器から分離して集中コントローラに置き、機器側はコントローラの指示どおりに転送するだけのデータプレーンに徹させる。コントローラと機器の間の代表的な標準プロトコルがOpenFlowで、コントローラは「この条件に合うパケットはこう処理せよ」というフローエントリ(マッチ条件+アクション)を機器のフローテーブルに書き込む。利点はネットワーク全体の一元管理と自動化——構成変更をソフトウェアから一括投入できる。一方で、コントローラが単一障害点になり得るため、その冗長化が設計論点になる。

NFV(Network Functions Virtualization)は別の軸の話だ。FW、ロードバランサ、ルータといった専用アプライアンスの「機能」を、汎用サーバ上の仮想マシンやソフトウェア(VNF)として動かす。ハードの調達・設置なしに機能を追加でき、性能もサーバ資源の増減で調整できる。複数のVNFを「FW→IDS→プロキシ」のように順に通す構成はサービスチェイニングと呼ばれ、通信の種類ごとに通す機能の並びをソフトウェアで定義できる。

両者は「ソフトウェアでネットワークを柔らかくする」仲間だが、対象が違う。SDNは制御の分離(誰が転送を決めるか)、NFVは機能の仮想化(何の上で機能が動くか)。この一文で区別できれば、午前IIの選択肢の入れ替え(SDNの説明にNFVの記述を混ぜる等)には引っかからない。なお、12-8節のEVPNのように「コントローラではなく標準プロトコル(BGP)で分散的に制御する」アプローチもあり、集中か分散かはアーキテクチャの選択である。

試験ではこう出る午前IIで「OpenFlowを用いたSDNの説明はどれか」→「コントローラがフローテーブルを介してスイッチの転送動作を集中制御する」。「NFVの説明はどれか」→「専用機器で実現していたネットワーク機能を汎用サーバ上のソフトウェアで実現する」。サービスチェイニングは「通信を複数の仮想ネットワーク機能に順序付けて通過させること」。定義文をそのまま選べるレベルで十分。

12-10IoTシステムのネットワークと5G — 2025年度午後IIテーマ

最後はIoTだ。2025年度午後IIでは「IoTシステムの設計」が出題された。IoTのネットワークは、これまでの「人が使うPC・スマホ」前提の設計と要件がまるで違う。デバイスが桁違いに多く(数千〜数万)、1台あたりのデータは小さく、電源・設置場所に制約があり、無人で長期間動く。この要件から各技術の存在理由を導けるようにしよう。

まず足回り。屋外に散らばるセンサを携帯電話網や無線LANで収容するのはコスト・電力の面で厳しい。そこでLPWA(Low Power Wide Area)——省電力・広域・低速に全振りした無線方式だ。通信速度は数十bps〜数百kbps程度と遅いが、電池で年単位の運用ができ、km単位の距離を飛ばせる。「速度を捨てて電力と距離を取る」というトレードオフの理解が本質で、細かい規格名より「センサの少量データを広域から集めるための無線」と押さえればよい。

次にプロトコル。大量のデバイスからデータを集めて複数のアプリケーションに配るとき、HTTPのような「クライアントがサーバに都度要求する」1対1モデルは効率が悪い。IoTの定番はMQTTで、パブリッシュ/サブスクライブ(出版/購読)という通信モデルを採る。

温度センサ×1000台            監視アプリ   分析基盤
 (パブリッシャ)             (サブスクライバ)(サブスクライバ)
     │                            ▲            ▲
     │①publish                   │③配信      │③配信
     │ topic: factory1/line2/temp │            │
     ▼                            │            │
  〔MQTTブローカ〕──────────┴────────┘
     ②topicを購読(subscribe)している相手に配信

図12-5: MQTTのパブリッシュ/サブスクライブモデル。送り手と受け手はブローカ経由で疎結合になり、互いの存在を知らなくてよい。

デバイス(パブリッシャ)はブローカと呼ばれる中継サーバへ、トピック(階層的な名前)を付けてメッセージを発行する。受け手(サブスクライバ)は関心のあるトピックをブローカに購読登録しておくと、該当メッセージが届き次第配信される。送り手と受け手が互いのアドレスを知る必要がない疎結合・1対多の構造なので、受け手のアプリを後から追加してもデバイス側は一切変更不要だ。ヘッダが小さく軽量で、非力なデバイスや細い回線に向く。到達保証のレベル(QoS)を0〜2から選べる点も特徴として押さえておく。

設計面の論点は3つ。第一にエッジコンピューティング——全データをクラウドに送ると回線帯域とクラウド処理費がかさみ、遅延も大きい。デバイスの近く(工場内のゲートウェイ等)で集約・前処理・即時判断を行い、クラウドへは要約だけ送る。第二にデバイス認証——数千台のデバイスにパスワード運用は non-starter で、クライアント証明書によるTLS相互認証が定石だ。台数分の証明書の発行・失効の管理(製造時の書き込み、漏えい時の失効)が設計課題として問われる。第三に網の分離——IoTデバイスは長寿命でパッチも当てにくいため、業務LANとはセグメントを分離し、通信先をブローカ等の必要最小限に絞る。

最後に5Gに一言。5Gは高速大容量だけでなく低遅延・多数同時接続を特徴とする移動通信で、試験の文脈ではローカル5G——通信事業者ではなく企業や自治体が自ら免許を受け、工場や港湾など限られたエリアに専用の5G網を構築する制度——がIoTの足回りとして登場する。「無線LANより広く安定したエリアを、自営で確保したい」場面の選択肢として覚えておけばよい。

例題 12-4工場内の1,000台のセンサが計測値を送信し、監視アプリと分析基盤の2システムがそれを利用する。次のア〜ウのうち、MQTTのパブリッシュ/サブスクライブモデルの説明として適切なものをすべて選べ。
(ア) センサは監視アプリと分析基盤それぞれのIPアドレス宛にデータを2回送信する。
(イ) センサはブローカへトピックを付けて1回発行すれば、購読中の2システムへ配信される。
(ウ) 利用システムを3つに増やす場合も、センサ側の設定変更は不要である。
解答と解説

正解: イ、ウ

パブリッシュ/サブスクライブの本質はブローカを介した疎結合。センサは宛先を知らず、ブローカへ1回publishするだけでよい(イ)。受け手の追加はブローカへの購読登録だけで完結し、送り手に影響しない(ウ)。アは1対1の直接送信モデル(HTTPでの直接POSTなど)の説明で、これだと受け手が増えるたびにセンサの送信回数と設定が増えていく——この対比を記述で説明できるようにしておこう。

LPWA=省電力・広域・低速MQTT=ブローカ経由の出版/購読エッジ=近くで前処理証明書=デバイス認証の定石ローカル5G=自営の5G網

Q1. 基幹システムのクラウド接続にインターネットVPNではなく閉域接続を選定する理由として、最も適切なものはどれか。
  • ア 導入コストが低く、開通までの期間も短いから
  • イ 帯域・遅延の品質保証があり、通信がインターネットを経由しないから
  • ウ SaaSを含むすべてのクラウドサービスへの通信が高速化されるから
  • エ 暗号化が不要になるため、端末側の処理負荷を削減できるから

正解: イ — 閉域接続の価値は「品質のSLA」と「閉域性(インターネット非経由)」の2軸(12-2節)。アは逆で、コストと開通の速さはインターネットVPNの利点。ウが紛らわしいが、SaaSは基本的にインターネット経由で提供されるため閉域接続の直接の対象ではない。エは一面の事実を含むが選定の主目的ではなく、閉域網でも重要通信の暗号化は行われ得る。

Q2. マルチクラウド構成を採用する主目的として、最も適切なものはどれか。
  • ア クラウド事業者間で通信を分散させ、利用料金を平準化する
  • イ 単一のクラウド事業者の大規模障害時にも、サービスを継続できるようにする
  • ウ 運用管理を単一の管理画面に統合し、運用負荷を削減する
  • エ オンプレミス設備を完全に廃止し、資産管理を不要にする

正解: イ — マルチクラウドの主目的は可用性向上であり、2023年度午後IIのテーマもこれだった(12-3節)。アの料金平準化は主目的ではない。ウは逆で、事業者が増えるぶん運用はむしろ複雑になる(マルチクラウドの代償)。エはハイブリッド/マルチの用語とも無関係で、オンプレ廃止はマルチクラウドの定義に含まれない。

Q3. ローカルブレイクアウトの説明として、適切なものはどれか。
  • ア 全拠点のインターネット向け通信を本社のプロキシに集約して検査する
  • イ 拠点間の通信をフルメッシュのVPNで直接接続する
  • ウ 特定のSaaS宛などの通信を、本社を経由させず拠点から直接インターネットへ送出する
  • エ SaaS宛の通信を閉域網経由に切り替えて品質を保証する

正解: ウ — ローカルブレイクアウトは「拠点から直接出す」こと(12-4節)。アはむしろ解決したい従来構成そのもの。イは拠点間WANの設計の話で、インターネット向けトラフィックの話ではない。エが紛らわしいが、SaaSは原則インターネット経由で提供されるため閉域網への切替という概念が成り立たない。

Q4. スパイン&リーフ構成の特徴として、適切なものはどれか。
  • ア STPにより冗長リンクをブロックし、ループのない木構造を保つ
  • イ すべてのリーフをすべてのスパインに接続し、ECMPで複数経路に負荷分散する
  • ウ リーフ同士を直結し、スパインを経由しない最短経路を確保する
  • エ コアスイッチにトラフィックを集約し、南北方向の通信を最適化する

正解: イ — 全リーフ×全スパインの接続とL3+ECMPによる全リンク活用が本質(12-5節)。アは従来の3階層L2設計の説明で、スパイン&リーフはSTPに頼らないことが利点。ウは誤りで、リーフ間通信は必ずスパインを経由する(だから等距離になる)。エは3階層構成の思想でEast-West中心の要件に合わない。

Q5. ライブマイグレーションを実行するために、移動元と移動先のホストに求められるネットワーク要件はどれか。
  • ア 両ホストが同一のL2セグメントに接続されており、VMのIPアドレスを変更せずに移動できること
  • イ 両ホストが異なるサブネットに属し、ルーティングで到達可能であること
  • ウ 移動後にVMへ新しいIPアドレスをDHCPで払い出せること
  • エ DNSレコードを書き換えて新ホストのアドレスへ誘導できること

正解: ア — 移動の前後でIP/MACを維持して通信を継続するのがライブマイグレーションの目的であり、同一L2セグメント(またはVXLAN等によるL2延伸)が要件になる(12-6節)。イはルーティング到達性だけではIP維持ができない。ウはIPが変わる時点でセッションが切れ、「ライブ」でなくなる。エはサーバ切替(12-3節のDNS切替)の手法で、稼働中VMの無停止移動には使えない。

Q6. VXLANのVNIで識別できるセグメント数はどれか。
  • ア 約4,000
  • イ 約65,000
  • ウ 約1,677万
  • エ 約43億

正解: ウ — VNIは24ビットで224=16,777,216、約1677万(12-7節)。アはVLAN ID(12ビット・実質4094個)で、VXLANが解決した側の数字。イは16ビット(65,536)、エは32ビット(IPv4アドレス空間)の数字で、いずれもビット数の混同を狙った選択肢。「VLAN=12ビット、VNI=24ビット」をセットで覚える。

Q7. VXLANにおけるVTEPの役割として、適切なものはどれか。
  • ア BUMトラフィックをアンダーレイ全体にブロードキャストで拡散する
  • イ 元のL2フレームへのVXLANヘッダの付与(カプセル化)と除去を行う、トンネルの出入口となる
  • ウ フローテーブルを配布してスイッチ群の転送を集中制御する
  • エ オーバーレイ区間の通信を暗号化し、盗聴を防止する

正解: イ — VTEP(VXLAN Tunnel End Point)はカプセル化/デカプセル化の出入口(12-7節)。アは誤り——BUMは同一VNIに参加するVTEPへの複製等で扱われ、アンダーレイ全体へのブロードキャストではない。ウはSDNコントローラの説明。エは誤りで、VXLANのカプセル化自体に暗号化機能はない(必要ならIPsec等を併用する)。

Q8. VXLANと組み合わせて使われるEVPNの役割として、適切なものはどれか。
  • ア L2フレームをUDP/IPでカプセル化して転送するデータプレーンを提供する
  • イ MP-BGPでMAC/IPの所在情報をVTEP間に広告し、フラッディングによる学習を削減する
  • ウ スパニングツリーを拡張し、オーバーレイのループを防止する
  • エ VTEP間のトンネルをIPsecで暗号化し、DC間通信を保護する

正解: イ — EVPNはVXLANのコントロールプレーンとして、BGPでMAC/IP経路を広告する(12-8節)。アが最も紛らわしいが、それはVXLAN自体(データプレーン)の役割——「データプレーン=VXLAN、コントロールプレーン=EVPN」の分担を問うのが出題の狙い。ウはSTPの話でEVPNとは無関係、エは暗号化機能でありEVPNの提供物ではない。

Q9. SDN(OpenFlowを用いた構成)の説明として、適切なものはどれか。
  • ア FWやロードバランサの機能を汎用サーバ上のソフトウェアとして実現する
  • イ コントロールプレーンを分離した集中コントローラが、フローテーブルを介してスイッチの転送動作を制御する
  • ウ L2フレームをL3ネットワーク上にトンネルで延伸し、仮想的なセグメントを構成する
  • エ 各スイッチが自律的に経路情報を交換し、分散的に転送先を決定する

正解: イ — SDNの本質は「コントロールプレーンとデータプレーンの分離+集中制御」で、OpenFlowはコントローラがフローエントリを書き込むための標準プロトコル(12-9節)。アはNFVの説明で、最も混同されやすい。ウはVXLANの説明。エは従来型の分散制御(OSPFやBGPを各機器が話す世界)で、SDNはむしろその対極にある。

Q10. MQTTの通信モデルの説明として、適切なものはどれか。
  • ア クライアントがサーバへ定期的にHTTPで問い合わせ、新着データを取得する
  • イ 送信元デバイスが受信側システムのIPアドレスを指定し、1対1で直接送信する
  • ウ パブリッシャがブローカへトピック付きでメッセージを発行し、そのトピックを購読するサブスクライバへ配信される
  • エ 送信元がセグメント内へブロードキャストし、必要なシステムだけが受信処理する

正解: ウ — MQTTはブローカを介したパブリッシュ/サブスクライブモデルで、送り手と受け手が互いを知らない疎結合・1対多を実現する(12-10節)。アはHTTPポーリングの説明で、大量デバイスでは無駄な問い合わせが激増する。イは直接送信モデルで、受け手の追加のたびにデバイス側の変更が要る。エはL2ブロードキャストの話で、IP網をまたぐIoT収容には成立しない。
