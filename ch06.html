基礎講座 第6章

DNS — 名前解決の設計と運用

DNSはNW試験で最も出題頻度の高い分野のひとつだ。単なる「名前→IPアドレスの変換」で終わらせず、ゾーンの委任・キャッシュの動き・サーバ移行の手順・攻撃と対策まで、設計と運用の言葉で語れるようになる。この章は他章より厚い。時間をかける価値がある。

学習目標:
  • 再帰問合せと反復問合せの違いを、登場する3種類のサーバとともに図解できる
  • 主要リソースレコード(A・CNAME・MX・NS・SOA・PTR)の役割と制約を説明できる
  • TTLを考慮したサーバ移行手順を設計できる
  • ゾーン転送(プライマリ/セカンダリ・シリアル番号)の仕組みを説明できる
  • オープンリゾルバとキャッシュポイズニングの危険性、その対策を記述できる

想定学習時間: 5〜6時間(例題・章末テスト込み)

6-1ドメイン名の階層とゾーン・委任

世界中のホスト名とIPアドレスの対応を1台のサーバで管理するのは不可能だ。数が膨大なうえ、更新が世界中で毎秒発生する。DNS(Domain Name System)の答えは階層化と分散管理——名前空間を木構造にして、枝ごとに管理を委ねる仕組みである。

ドメイン名は右から読むと木構造そのものだ。www.example.jpなら、頂点のルート(名前上は末尾の「.」)の下にjp(トップレベルドメイン)、その下にexample.jp、その中のホストwwwという階層になる。そして各階層の管理者は、配下の部分木の管理を別の組織に委任できる。jpの管理者はexample.jpの中身を知らない。「example.jpのことはこのネームサーバに聞け」という委任情報(NSレコード)だけを持つ。この「自分で管理する範囲」の単位をゾーンと呼ぶ。ドメインが「名前空間上の領域」を指すのに対し、ゾーンは「委任によって切り出された管理責任の単位」だ。

                     ┌─────┐
                     │ ルート │  ゾーン: ルート
                     └──┬──┘  「jpのことはjpのDNSへ」= 委任
              ┌────────┼────────┐
            [jp]      [com]     [org] …  ゾーン: jp
              │く「example.jpのことは ns1.example.jp へ」= 委任
        ┌─────┴─────┐            +グルーレコード(ns1のIPアドレス)
   [example.jp]  [ipa.go.jp]…    ゾーン: example.jp
        │                        (ここは自組織で管理)
   www / mail / ns1 …

図6-1: ドメイン名の木構造と委任。各ゾーンの管理者は配下への「案内板」(NSレコード)だけを持つ。

委任には1つ、パズルのような問題がある。jpゾーンが「example.jpのことはns1.example.jpに聞け」と案内するとき、そのns1.example.jpのIPアドレスはexample.jpゾーンの中にある。つまり「example.jpに聞くためにexample.jpに聞く」という循環が生じてしまう。これを断ち切るため、親ゾーン(jp)側に、委任先ネームサーバのIPアドレス(Aレコード)を例外的に置いておく。これがグルーレコード(glue record)——「糊」の名のとおり、親と子のゾーンを貼り合わせる情報だ。委任先サーバの名前が委任されるゾーンの内部にある場合に必要になる、という条件まで押さえておくと午前IIで確実に得点できる。

合格の視点DNSの全体像は「木構造+案内板のリレー」に尽きる。この後の再帰・反復問合せ(6-2)もゾーン転送(6-5)も、すべて「どのサーバがどのゾーンの案内板を持つか」の話として整理できる。図6-1を白紙に描き直せるようにしてから先へ進もう。

6-2名前解決の登場人物 — 再帰問合せと反復問合せ

名前解決には役割の異なる3種類のプレーヤが登場する。この3者の区別がDNS理解の土台であり、試験でも混同を誘う問題が繰り返し出る。

  • スタブリゾルバ — PCやサーバのOSに組み込まれた問合せ機能。自分では解決の旅をせず、設定されたフルリゾルバに「答えが出るまでやって、結果だけください」と依頼するだけの存在。
  • フルリゾルバ(フルサービスリゾルバ、キャッシュDNSサーバ) — スタブリゾルバの依頼を受け、ルートから権威サーバを順にたどって答えを見つけ出す働き者。得た答えを一定時間キャッシュして、次の問合せに高速に答える。社内DNSサーバやISPのDNSサーバがこれ。
  • 権威DNSサーバ(コンテンツサーバ) — 特定のゾーンの情報(ゾーンファイル)を正式に保持し、そのゾーンについての問合せに答えるサーバ。自分の知らない名前について解決の旅をすることはない。

そして問合せには2つの流儀がある。スタブリゾルバがフルリゾルバへ送るのが再帰問合せ——「最終的な答えを返してほしい」という丸投げの依頼。フルリゾルバが各権威サーバへ送るのが反復問合せ——権威サーバは自分の知る範囲で「答え」または「次に聞くべきサーバの案内」を返し、フルリゾルバがそれを頼りに次々と聞いて回る。依頼は再帰、旅は反復。図で流れを固めよう。

[PC]              [フルリゾルバ]        [ルート]  [jp権威]  [example.jp権威]
 スタブリゾルバ      (キャッシュDNS)
   │ ①再帰問合せ        │
   │ www.example.jp は? │
   │ ─────────────→   │ ②反復問合せ ────→ │
   │                    │ ←「jpはこちら」── │
   │                    │ ③反復問合せ ──────────→ │
   │                    │ ←「example.jpはこちら」── │
   │                    │ ④反復問合せ ────────────────→ │
   │                    │ ←「www = 192.0.2.20」(権威応答)── │
   │ ⑤答えを返す        │
   │ ←─────────────   │ ⑥結果をTTLの間キャッシュ
   │                    │    (次回は②〜④を省略して即答)

図6-2: 名前解決のシーケンス。①が再帰問合せ、②〜④が反復問合せ。フルリゾルバだけが「旅」をする。

ここで大事な性質が2つある。第一に、フルリゾルバは②〜④で得た情報(途中の委任情報も含む)をキャッシュするので、2回目以降の同じ問合せ、あるいは同じjpドメイン配下の別名の問合せは大幅に短縮される。DNSの応答速度と、ルートサーバ群が世界の問合せに耐えられる理由は、このキャッシュの多段の積み重ねにある。第二に、どのサーバの応答かで答えの重みが違う。権威サーバからの応答は権威応答(オーソリティのある答え)、フルリゾルバがキャッシュから返す答えは非権威応答と呼ばれ、nslookupの表示にも「Non-authoritative answer」として現れる(6-10)。

例題 6-1社内PCがブラウザに www.example.jp を入力してから名前解決が完了するまでの動作ア〜カを、正しい順に並べよ。ただし各サーバのキャッシュは空とする。
ア フルリゾルバがjpの権威サーバに問い合わせ、example.jpの権威サーバの案内を得る
イ PCのスタブリゾルバが社内のフルリゾルバに再帰問合せを送る
ウ フルリゾルバがexample.jpの権威サーバに問い合わせ、wwwのIPアドレス(権威応答)を得る
エ フルリゾルバがルートサーバに問い合わせ、jpの権威サーバの案内を得る
オ フルリゾルバが結果をキャッシュし、PCに答えを返す
カ PCがOSのDNS設定(参照先サーバ)を確認する
解答と解説

カ → イ → エ → ア → ウ → オ

スタブリゾルバはまず自分の参照先(カ)を確認して再帰問合せを投げ(イ)、以降の旅はフルリゾルバの反復問合せで、木構造を上から下へ——ルート(エ)→jp(ア)→example.jp(ウ)——とたどる。答えはキャッシュされてからPCへ(オ)。並べ替えで迷ったら「依頼は1回、旅は上から下へ」と唱えること。この流れの並べ替え・穴埋めは午後の定番の導入設問だ。

試験ではこう出る午前IIの定番は「再帰問合せと反復問合せの説明の組合せ」。スタブ→フルリゾルバが再帰、フルリゾルバ→権威が反復という対応を逆にした選択肢が必ず並ぶ。また「キャッシュDNSサーバ」「コンテンツサーバ」という呼び名でも出るので、フルリゾルバ=キャッシュDNS、権威=コンテンツの同義語ペアも頭に入れておこう。

6-3リソースレコード詳解 — ゾーンファイルを読む

権威サーバが持つゾーンの中身は、リソースレコード(RR)の集まりだ。レコードは「名前 TTL クラス タイプ 値」の形式で書かれる(クラスは事実上INのみ)。まず実物のゾーンファイルの例を見てから、タイプごとに役割を整理しよう。

$ORIGIN example.jp.        ; 以降の相対名の基準
$TTL 86400                 ; 既定TTL(秒)= 24時間

@       IN SOA  ns1.example.jp. admin.example.jp. (
                2026070401 ; シリアル番号(版数)
                7200       ; リフレッシュ間隔
                900        ; リトライ間隔
                1209600    ; 有効期限(エクスパイア)
                300 )      ; ネガティブキャッシュTTL
@       IN NS   ns1.example.jp.
@       IN NS   ns2.example.jp.
@       IN MX   10 mail.example.jp.
@       IN MX   20 mail2.example.jp.
ns1     IN A    192.0.2.1
ns2     IN A    198.51.100.1
mail    IN A    192.0.2.10
mail2   IN A    198.51.100.10
www     IN A    192.0.2.20
shop    IN CNAME www.example.jp.
@       IN TXT  "v=spf1 mx -all"

図6-3(コード例): example.jpゾーンのゾーンファイル。@はゾーン名(example.jp.)自身を表す。この形式の穴埋めが午後でそのまま出る。

タイプ意味試験で問われるポイント
A / AAAA名前→IPv4 / IPv6アドレスDNSの主役。1つの名前に複数のAを書ける(→6-9のラウンドロビン)
CNAME名前の別名(エイリアス)。正式名を指す共存の制約が頻出(本文参照)
MXそのドメイン宛メールの受取サーバプリファレンス値は小さいほど優先。値にはホスト名を書く(IPアドレス直書き不可)
NSゾーンのネームサーバ(委任情報)親ゾーン側と子ゾーン側の両方に書かれる
SOAゾーンの管理情報の要シリアル番号(6-5)、ネガティブキャッシュTTL
PTRIPアドレス→名前(逆引き)in-addr.arpa配下。192.0.2.10の逆引き名は10.2.0.192.in-addr.arpa(オクテット逆順)
TXT任意のテキストSPFなどメール送信ドメイン認証に使う(第7章)

表6-1: 主要リソースレコード。「MXは小さい値が勝ち」「PTRは逆順」の2つはうっかりミスの多発地帯だ。

CNAMEには重要な制約がある。CNAMEを付けた名前には、原則として他のレコードを共存させられない。「shopはwwwの別名だ」と宣言した以上、shopに独自のMXやAがあると「別名なのに独自の属性を持つ」矛盾が生じるからだ。この帰結として、(1) ゾーンの頂点(example.jp.自身)はSOAとNSを必ず持つためCNAMEにできない、(2) MXやNSの値にCNAMEの名前を指定すべきでない(参照先が別名だと解決が二度手間になり、動作が不安定になる実装もある)、という2つの試験頻出ルールが導かれる。

例題 6-2図6-3のゾーンファイルについて答えよ。
(1) example.jp宛のメールは、通常時どのサーバ(ホスト名)に配送されるか。mail2はどのような場合に使われるか。
(2) 「@ IN MX 10 192.0.2.10」という書き方が誤りである理由を述べよ。
(3) 新サービス用に「shop.example.jp にも独自のメールサーバを立てたい」と考え、shopにMXレコードを追加しようとしたが、設定できない。理由を述べよ。
解答と解説

(1) プリファレンス値が最小(10)のmail.example.jpに配送される。mail2(値20)は、mailに接続できないときのバックアップ(代替受取先)として使われる。値の大小と優先順位が逆に感じられる人が多い——「距離のようなもので、近い(小さい)ほうから試す」と覚えると間違えない。

(2) MXレコードの値にはホスト名を指定する決まりであり、IPアドレスを直接書くことはできないから。正しくは「MX 10 mail.example.jp.」とし、mailのIPアドレスは別途Aレコードで定義する。MXとAの2段構えという関係そのものが穴埋めの題材になる。

(3) shopには既にCNAMEレコードがあり、CNAMEを持つ名前には他のレコードを共存させられないから。独自のMXを持たせたければ、CNAMEをやめてshopをAレコード(正式名)に変更する必要がある。

6-4TTLとキャッシュ — サーバ移行の定番手順

各リソースレコードにはTTL(Time To Live)が付いており、フルリゾルバは受け取った答えをこの秒数だけキャッシュしてよい。TTLが86400(24時間)なら、権威サーバでレコードを書き換えても、最悪24時間は世界中のキャッシュに古い値が残り続ける。TTLは「応答の速さ・権威サーバの負荷軽減」と「変更の伝わりやすさ」のトレードオフを決めるつまみだ。長くすればキャッシュがよく効き、短くすれば変更がすぐ行き渡る。

この性質が最も問われるのがサーバ移行(IPアドレス変更)の場面だ。何も考えずにAレコードを書き換えると、TTLが切れるまで新旧のアドレスにアクセスが分かれてしまう。そこで定番の手順がある。

  1. 移行のTTL(旧TTL)以上前に、TTLを短縮する — 例: 86400秒→300秒。ここで注意。TTL短縮の変更自体も、既にキャッシュされた「TTL=86400の古い答え」が切れるまで行き渡らない。だから切替予定の少なくとも旧TTL(24時間)前までに短縮を済ませておく必要がある。
  2. 切替え — Aレコードを新IPアドレスに書き換える。TTLは300秒なので、遅くとも5分後には世界中の問合せが新サーバへ向かう。切替直後の混在期間も5分で済む。
  3. 確認後、TTLを元に戻す — 新サーバでの稼働を確認したら、TTLを86400秒に戻してキャッシュ効率を回復する。問題があれば旧IPに戻す判断も5分で反映できる——TTL短縮は「切戻しを速くする保険」でもある。

もうひとつ、SOAレコード最後の値ネガティブキャッシュTTLにも触れておこう。「その名前は存在しない(NXDOMAIN)」という否定の答えもキャッシュされる。新しいレコードを追加する直前にその名前を引いてしまうと、「存在しない」がキャッシュされ、追加後もしばらく引けない——という落とし穴の原因になる。

例題 6-3www.example.jp(Aレコード、TTL=86400秒)のWebサーバを、7月10日の10:00に新IPアドレスへ切り替えることになった。
(1) TTLを300秒に短縮する作業は、遅くともいつまでに行うべきか。理由とともに答えよ。
(2) 10:00にAレコードを書き換えた場合、全ての利用者のアクセスが新サーバに向かうのは遅くとも何時何分か。
(3) 切替後もTTLを300秒のままにせず86400秒に戻すのはなぜか。
解答と解説

(1) 7月9日の10:00まで(切替の86400秒=24時間前まで)。TTL短縮という変更自体が、キャッシュ済みの旧TTL(86400秒)の答えが期限切れになるまで各フルリゾルバに反映されないから。ぎりぎりに短縮しても、古い「TTL=86400」の答えを持つキャッシュには効かない——この理由を書かせるのが出題の核心だ。

(2) 10:05。切替時点で各キャッシュが持つ答えのTTLは最大300秒。10:00直前にキャッシュした利用者でも、300秒後の10:05には期限が切れて新しい答えを取りにいく。

(3) TTLが短いままだとキャッシュの効果が薄れ、フルリゾルバから権威サーバへの問合せが増えて権威サーバの負荷と応答遅延が増大するから。移行が安定したらキャッシュ効率を回復させるのが定石だ。

試験ではこう出るTTL事前短縮は午後の記述で最も「型どおり」に出るテーマのひとつ。問われ方は「切替前に行っておくべきDNSの設定変更とその時期を述べよ」。解答の骨子は「切替の旧TTL以上前に、対象レコードのTTLを短い値に変更しておく。キャッシュに残る旧TTLの応答が消えるまで短縮が効かないため」。時期の根拠まで書けて満点になる。

6-5ゾーン転送 — プライマリとセカンダリの同期

権威サーバが1台では、故障した瞬間にそのゾーンの名前解決が世界から失敗し始める。そこで権威サーバは複数台で運用するのが原則で、ゾーンファイルを編集するプライマリサーバと、その複製を持つセカンダリサーバに役割を分ける。外から見ればどちらも対等な権威サーバであり、NSレコードに両方を登録して問合せを分散する。

複製の同期がゾーン転送だ。仕組みの鍵はSOAレコードのシリアル番号にある。

  • セカンダリはSOAのリフレッシュ間隔ごとにプライマリのSOAを確認し、シリアル番号が自分の保持する値より大きければ「更新があった」と判断してゾーン転送を要求する。だからゾーンを編集したら必ずシリアル番号を増やす。増やし忘れるとセカンダリは永遠に古いまま——DNS運用の古典的な事故であり、試験の障害問題の題材でもある。
  • 転送には、ゾーン全体を送るAXFR(完全ゾーン転送)と、差分だけ送るIXFR(差分ゾーン転送)がある。大きなゾーンではIXFRが転送量を大幅に節約する。
  • リフレッシュ間隔を待つと更新の反映が遅れるため、プライマリが更新時にNOTIFYメッセージをセカンダリへ送って「変わったよ、取りに来て」と即時に促す仕組みがある。
  • プライマリに長期間アクセスできない場合、セカンダリはエクスパイア(有効期限)を過ぎた時点でゾーンの応答を停止する。古すぎる情報で答え続けるより、答えないほうがましという設計だ。

トランスポートにも注目しておこう。通常の名前解決の問合せはUDPポート53が基本(応答が大きい場合はTCPへ切替え)だが、ゾーン転送はデータ量が多く確実性が必要なためTCPポート53を使う。ファイアウォールの設定問題で「DNSはUDP/53だけ許可」と書かれていたら、ゾーン転送が失敗する伏線だ——この形の出題は繰り返し登場している。あわせて、ゾーン転送の要求元をセカンダリのIPアドレスに限定する設定も重要になる。誰にでもAXFRを許すと、ゾーン内の全ホスト情報(内部構成)が丸ごと第三者に渡ってしまうからだ。

合格の視点ゾーン転送は「シリアル番号で版数管理された、プライマリ→セカンダリの一方向レプリケーション」と要約できる。シリアル番号・NOTIFY・AXFR/IXFR・TCP 53・転送元制限——この5点セットで、午前IIも午後の設計記述もカバーできる。

6-6内部DNSと外部DNSの分離設計

企業のDNSをどう配置するかは、NW午後試験の構成図に必ずと言っていいほど現れる設計テーマだ。原則は明快で、役割の異なるDNSを1台に同居させない。具体的には2つの軸で分離する。

第一の軸は公開する情報の分離(内部DNSと外部DNS)。社外に見せてよいのは公開サーバ(www、メールのMX)の名前だけだ。社内のファイルサーバや業務システムの名前・IPアドレスは、攻撃者にとって格好の下調べ材料(内部構成の地図)になるため、外から引けてはならない。そこで、インターネットに公開する外部DNS(権威)には公開ゾーンだけを載せてDMZに置き、社内専用の名前を管理する内部DNSは社内ネットワークに置いて社外からのアクセスを遮断する。同じexample.jpという名前空間でも、外向けには最小限のレコード、内向けには全レコード、と見る場所によって答えを変える構成になる。

第二の軸は機能の分離(権威サーバとフルリゾルバ)。外部DNSは「うちのゾーンを世界に答える」権威専用とし、再帰問合せは受け付けない。逆に社内PCのための名前解決(フルリゾルバ機能)は内部側に置き、社内からの問合せだけを受け付ける。この機能分離が、次節のオープンリゾルバ問題とキャッシュポイズニング対策(6-8)の土台になる。

インターネット
      │
      │  ①社外の誰か → 外部DNSへ「www.example.jpは?」(権威応答のみ)
   ┌──┴───────────── DMZ ─────────────┐
   │ [外部DNS(権威専用)]  [公開Web] [メール] │
   │   ・公開ゾーンのみ保持                    │
   │   ・再帰問合せ拒否                        │
   └──┬────────────────────────────┘
     [FW]
   ┌──┴──────────── 社内LAN ────────────┐
   │ [内部DNS(権威+フルリゾルバ)]              │
   │   ・社内ゾーン(業務サーバ等)を保持          │
   │   ・社内からの再帰問合せを受付け              │
   │   ②社内PC → 内部DNS → (社外の名前は        │
   │      フォワーダ経由等でインターネット側へ解決) │
   │ [PC] [PC] [業務サーバ]                      │
   └───────────────────────────────┘

図6-4: 内部・外部DNS分離の典型構成。「外部=権威専用・公開情報のみ」「内部=社内専用・外から到達不可」が設計の芯。

この構成では、社内PCが社外の名前(例: www.ipa.go.jp)を引くとき、内部DNSが自らインターネットへ反復問合せするか、指定した転送先(フォワーダ)へ問合せを委ねる。いずれにせよ社内→外への問合せ経路と、外→DMZの権威への問合せ経路は別物として構成図を読み解くこと。午後問題では「このFWのルールでどのDNS通信が通るか」を追わせる設問が定番で、①と②の矢印を混同すると芋づる式に失点する。

試験ではこう出る「内部DNSと外部DNSを分離する目的を述べよ」は記述の頻出テーマ。解答の芯は「社内ホストの名前・アドレスなど内部構成の情報を社外から取得できないようにするため」。機能面の分離を問われたら「権威機能と再帰(キャッシュ)機能を分け、外部公開サーバでは再帰問合せを受け付けないため」。Webシステム更改(DMZ/DNS/FW)という形で過去の午後問題でも扱われた、本命中の本命だ。

6-7オープンリゾルバの危険 — DNSリフレクション攻撃

フルリゾルバ(キャッシュDNS)を、送信元を制限せず誰からの再帰問合せにも答える状態で公開してしまったものをオープンリゾルバと呼ぶ。一見「親切なサーバ」だが、これはDDoS攻撃の踏み台そのものだ。

悪用の手口がDNSリフレクション(リフレクタ)攻撃である。仕組みは3つの部品でできている。

  1. 送信元IPアドレスの詐称 — DNSの問合せは基本UDPなので、コネクション確立が不要。攻撃者は送信元IPを攻撃対象のアドレスに偽装した問合せを送れる。
  2. 反射 — オープンリゾルバは問合せに律儀に答える。その応答は、詐称された送信元=攻撃対象に向かって飛ぶ。攻撃者の身元は隠れ、リゾルバが「反射板」にされる。
  3. 増幅 — DNSは小さな問合せ(数十バイト)に対して大きな応答(数千バイト。サイズの大きいレコードを問い合わせる)を返し得る。攻撃者が使った帯域の何十倍ものトラフィックが対象に集中する。増幅(アンプ)攻撃と呼ばれるゆえんだ。

対策は発想が明快で、「反射板にされない」ことに尽きる。再帰問合せを受け付ける送信元を自組織(社内・自ISP契約者)に限定するのが第一。6-6の設計がそのまま効いてくる——権威サーバは再帰を受け付けず、フルリゾルバは内側からしか使えなくする。加えて、ネットワークの出口で送信元IPが詐称されたパケットをそもそも外に出さないフィルタリング(送信元検証)をISPや組織が行うことも、インターネット全体としての対策になる。自分が攻撃されていなくても、設定の甘いサーバが加害に使われる——「オープンリゾルバにしない」のは公衆衛生に近い運用責務だ。

合格の視点リフレクション攻撃の記述問題は「UDPだから詐称できる」「応答が問合せより大きいから増幅する」の2点を含めると論理が完結する。NTPやmemcachedなど他のUDPサービスでも同型の攻撃が成立する、と一般化して理解しておくと応用が利く(セキュリティ全般は第8章)。

6-8キャッシュポイズニングの原理と対策

フルリゾルバのキャッシュに偽の情報を注入する攻撃がDNSキャッシュポイズニングだ。成功すると、そのリゾルバを使う全利用者が、正しい名前を引いたつもりで攻撃者のサーバ(偽サイト・偽メールサーバ)へ誘導される。利用者側の見た目は完全に正常なので、フィッシングよりたちが悪い。

攻撃の原理を理解するには、リゾルバが「応答の本人確認」を何でやっているかを知る必要がある。フルリゾルバは反復問合せをUDPで送り、返ってきたパケットが正当な応答かどうかを、(1) 送信元IPが問合せ先の権威サーバか、(2) 宛先ポートが問合せ時の送信元ポートと一致するか、(3) ヘッダのトランザクションID(16ビットの乱数)が一致するか、(4) 質問内容が一致するか——で判定する。攻撃者はここを突く。本物の権威サーバの応答が届くより先に、権威サーバのIPを詐称した偽応答を大量に送り込み、トランザクションIDなどの照合値を当てにいく。競争(レース)に一度でも勝てば、偽レコードがTTLの間キャッシュに居座る。かつてリゾルバの送信元ポートが固定されていた時代は、当てるべき値が実質16ビットのIDだけで、総当たりが現実的に成功してしまった。さらに、存在しないランダムなサブドメインを次々問い合わせさせて、キャッシュに邪魔されずに何度でもレースを仕掛ける手法(カミンスキー型攻撃)の登場で、この攻撃は「理論上の話」から「現実の脅威」になった。

対策は多層で考える。

  • 送信元ポートのランダム化 — 問合せごとに送信元ポートを乱数にすれば、攻撃者はトランザクションID(16ビット)に加えてポート番号も当てなければならず、当てるべき組合せが桁違いに増えて成功確率が激減する。現在のリゾルバの標準動作だが、NAPT機器がポートを変換し直してランダム性を打ち消してしまう構成には注意が必要で、これも試験の題材になる。
  • DNSSEC — 根本対策。権威サーバがレコードにディジタル署名を付け(RRSIG)、リゾルバが公開鍵(DNSKEY)で検証する。鍵の正しさは、親ゾーンに登録したハッシュ(DS)を介してルートから連なる信頼の連鎖で保証する。署名が検証できない偽応答はキャッシュされない。つまりDNSSECが提供するのは出自の認証と完全性(改ざん検知)であり、通信の暗号化(機密性)ではない——ここが最頻出の引っかけだ。
  • 運用面 — リゾルバを再帰制限する(6-7)、キャッシュを外から覗かれ・汚されにくくする機能分離(6-6)。ポイズニングの被害面を減らす意味でも分離設計が効く。
例題 6-4キャッシュポイズニング攻撃について答えよ。
(1) フルリゾルバの問合せの送信元ポートをランダム化すると攻撃が成功しにくくなる理由を、攻撃者が偽応答を受け入れさせる条件に触れて述べよ。
(2) DNSSECを導入すると偽応答がキャッシュされなくなる仕組みを、「署名」「検証」の語を用いて述べよ。
(3) 「DNSSECを導入すれば問合せ内容の盗聴も防げる」という主張の誤りを指摘せよ。
解答と解説

(1) 偽応答が受け入れられるには、トランザクションID(16ビット)に加え、問合せの送信元ポート宛に応答を届ける必要がある。ポートが固定なら攻撃者はIDだけを総当たりすればよいが、ランダム化するとIDとポートの組合せを同時に当てる必要が生じ、本物の応答が届くまでの短時間に的中させることが確率的に困難になるから。

(2) 権威サーバがゾーンのレコードに秘密鍵でディジタル署名を付与し、リゾルバが対応する公開鍵で署名を検証する。攻撃者は正しい秘密鍵を持たず有効な署名を偽造できないため、検証に失敗する偽応答は破棄され、キャッシュに登録されない。公開鍵自体の正しさは親ゾーンからの信頼の連鎖(DSレコード)で保証される。

(3) DNSSECは応答の出自の認証と改ざん検知(完全性)を提供する技術であり、通信を暗号化しない。問合せ・応答の内容は平文のまま流れるため、盗聴(機密性の侵害)は防げない。「認証・完全性」と「機密性」の取り違えを突く出題は午前II・午後とも定番だ。

6-9DNSラウンドロビンと負荷分散の限界

1つの名前に複数のAレコードを登録すると、権威サーバ・フルリゾルバは応答のたびにレコードの順序を入れ替えて返す。多くのクライアントは先頭のアドレスを使うため、結果としてアクセスが複数サーバに分散する。これがDNSラウンドロビン——追加投資ゼロで始められる、最も手軽な負荷分散だ。

ただし、その手軽さと引き換えに構造的な限界を抱えている。試験では「ラウンドロビンで十分か」を判断させる形で問われるので、限界を列挙できるようにしておく。

  • 死活を知らない — DNSはサーバの稼働状態を確認しない。3台中1台が故障しても、その故障サーバのアドレスを約1/3の利用者に配り続ける。障害時に自動で切り離せないのが最大の弱点。
  • キャッシュが分散を歪める — 応答はTTLの間フルリゾルバにキャッシュされる。大企業のリゾルバがキャッシュした1つの答えを、その配下の数千人が使い続けるため、意図した比率では分散しない。TTLを短くすれば緩和できるが、問合せ負荷が増える。
  • 均等にしか配れない — サーバの性能差に応じた重み付けや、接続数を見た動的な振り分けはできない。
  • セッションを追えない — 同じ利用者を同じサーバに固定する(セッション維持)制御ができない。

これらの限界を超えるには、専用の仕組みが要る。サーバの前段に置いてヘルスチェックと重み付き振り分けを行う負荷分散装置(LB)、DNSの応答自体を死活監視・拠点の距離などと連動させて変えるGSLB(広域負荷分散)、そしてコンテンツを世界中に配るCDN——それぞれの詳細は第7章と第9章で扱う。ここでは「DNSラウンドロビンの限界を言えること」がそれらの技術の存在理由の説明になる、という接続を押さえておけばよい。

試験ではこう出る「DNSラウンドロビンによる負荷分散の問題点を述べよ」の模範筋は「サーバの死活を確認しないため、障害サーバへも問合せ結果が配られ続ける」。もう一つ挙げるなら「応答がキャッシュされるため分散の比率を制御できない」。この2つを書ければ十分に合格答案だ。

6-10トラブルシューティング — nslookupとdigの読み方

DNSの障害切り分けは、この章で学んだ登場人物のどこで話が止まっているかを特定する作業だ。道具はnslookup(Windows標準)とdig(Unix系の定番)。試験でも実行結果の断片が示され、「この出力から何が分かるか」を読み取らせる。

$ dig www.example.jp A

;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 31742
;; flags: qr rd ra; QUERY: 1, ANSWER: 1, ...

;; QUESTION SECTION:
;www.example.jp.        IN  A

;; ANSWER SECTION:
www.example.jp.  280    IN  A  192.0.2.20
                 ~~~
                  TTL(キャッシュ応答なら問い合わせるたび減っていく)

;; SERVER: 10.1.1.53#53(10.1.1.53)   ← 誰に聞いたか

図6-5: digの出力の読みどころ。status、ANSWERセクション、TTL、問合せ先サーバの4点をまず見る。

読み方の要点を、切り分けの手順として整理する。

  • status(応答コード)を見る — NOERRORは正常応答。NXDOMAINは「その名前は存在しない」という権威側からの明確な否定(レコードの登録漏れ・タイプミスを疑う)。SERVFAILは「解決処理に失敗した」(問合せ先より先の障害、ゾーン転送切れでセカンダリが応答停止、DNSSEC検証失敗などを疑う)。NXDOMAINとSERVFAILの区別は、原因が「登録の問題」か「経路・サーバの問題」かの分水嶺だ。
  • TTLの減り方を見る — 同じ問合せを繰り返してTTLが減っていくなら、その答えはフルリゾルバのキャッシュから返っている。古い値が返り続けるときは、TTL満了までキャッシュに残る旧情報(6-4)を疑う。
  • 聞く相手を変えて切り分ける — 既定のリゾルバで失敗したら、dig @権威サーバのIP www.example.jpのように権威サーバへ直接聞いてみる。権威が正しく答えるならフルリゾルバ側(キャッシュ・経路・FW)の問題、権威も答えないならゾーン側の問題、と切り分けられる。nslookupならnslookup www.example.jp 権威サーバのIPと第2引数で指定する。
  • 非権威応答の表示に注意 — nslookupの「Non-authoritative answer」は異常ではなく、キャッシュから答えたという意味。権威応答かどうかは、digならヘッダのaaフラグの有無で分かる。

この章で追いかけてきた「スタブ→フルリゾルバ→権威」という一本道を、上流から順に疑っていく——DNSトラブルシューティングは結局、6-2の図をなぞる行為に他ならない。手元の環境でdigやnslookupを実際に打ち、自宅のリゾルバや公開されている権威サーバの応答を眺めておくと、出力形式は自然に頭に入る。読むだけで覚えようとしないことだ。

チェックテスト(10問)

DNSは最頻出分野。10問すべて根拠を言えるまで繰り返そう。

Q1. DNSの名前解決における問合せの説明として、適切なものはどれか。
  • ア スタブリゾルバは、ルートサーバから順に権威サーバをたどる反復問合せを行う
  • イ フルリゾルバは、スタブリゾルバから再帰問合せを受け、権威サーバへは反復問合せを行う
  • ウ 権威サーバは、自分の知らない名前について他の権威サーバへ再帰問合せを行う
  • エ フルリゾルバは、権威サーバへ再帰問合せを送り、最終的な答えだけを受け取る

正解: イ — 「依頼(再帰)を受けるのがフルリゾルバ、旅(反復)をするのもフルリゾルバ」。アは主語が誤りで、旅をするのはスタブではなくフルリゾルバ。ウは誤り——権威サーバは自ゾーンに答えるだけで解決の旅はしない。エは再帰と反復を逆にした定番の引っかけだ。

Q2. グルーレコードの説明として、適切なものはどれか。
  • ア 委任先ゾーンの内部にあるネームサーバ名のIPアドレスを、親ゾーン側に登録したAレコード
  • イ プライマリとセカンダリのゾーン内容を同期させるためのレコード
  • ウ CNAMEの参照先の名前を解決するために子ゾーン側に置くレコード
  • エ 逆引き用にin-addr.arpaゾーンへ登録するPTRレコード

正解: ア — 「example.jpのネームサーバはns1.example.jp」という委任では、ns1のアドレスを知るためにexample.jpへ聞く循環が生じる。これを断つため親(jp)側にns1のAレコードを置く——これがグルーレコード。委任先サーバの名前が委任されるゾーン内にある場合に必要、という条件つきの定義まで書けると強い。イはゾーン転送(シリアル番号)の話、エはPTRの一般的説明で、いずれも「親ゾーン側に置く」というグルーの本質と無関係だ。

Q3. CNAMEレコードに関する制約の説明として、適切なものはどれか。
  • ア CNAMEの参照先には、IPアドレスを直接指定しなければならない
  • イ 1つのゾーンに登録できるCNAMEは1つまでである
  • ウ CNAMEを設定した名前には、原則として他のレコードを共存させられない
  • エ CNAMEはゾーンの頂点(ゾーン名自身)にのみ設定できる

正解: ウ — 「この名前は別名」と宣言した以上、同じ名前が独自の属性(AやMX)を持つと矛盾するため共存不可。この帰結として、SOA・NSを必ず持つゾーン頂点はCNAME化できない——エは正反対の記述だ。アは誤りで、参照先は正式名(ホスト名)。イのような個数制限はない。

Q4. あるゾーンに「MX 10 mail1.example.jp.」と「MX 20 mail2.example.jp.」が登録されている。メール配送の動作として適切なものはどれか。
  • ア プリファレンス値の大きいmail2が優先して使われる
  • イ プリファレンス値の小さいmail1が優先され、接続できないときmail2が使われる
  • ウ 2台へ交互に配送され、負荷が均等化される
  • エ mail1へ配送された後、mail2へも同じメールが複製される

正解: イ — MXのプリファレンスは小さいほど優先。値20のサーバは値10のサーバに接続できない場合のバックアップだ。アは大小を逆にした最頻出の引っかけ。ウのような分散は同一値のMXを複数書いた場合に近い動きになるが、値が異なる本問では起きない。エのような複製配送の仕組みはMXにはない。

Q5. プライマリDNSサーバでゾーンファイルを編集したのに、セカンダリサーバの内容がいつまでも更新されない。最も疑うべき原因はどれか。
  • ア SOAレコードのシリアル番号を増やし忘れた
  • イ ネガティブキャッシュTTLが長すぎる
  • ウ セカンダリへのNOTIFYが遅延している
  • エ IXFRではなくAXFRが使われている

正解: ア — セカンダリは「シリアル番号が自分より大きいか」だけで更新の有無を判断する。増やし忘れれば、リフレッシュ間隔ごとに確認しても「更新なし」と判定し続ける。ウのNOTIFYは即時性を高める補助であり、なくてもリフレッシュ間隔で追いつくはずだから「いつまでも更新されない」の説明にならない。イは否定応答のキャッシュの話、エは転送方式の違いにすぎず、どちらも同期の成否そのものには影響しない。

Q6. Aレコード(TTL=86400秒)のIPアドレスを切り替える保守作業で、TTLを300秒へ短縮する変更を行う時期として適切なものはどれか。
  • ア 切替えと同時に行う
  • イ 切替えの300秒以上前までに行う
  • ウ 切替えの86400秒以上前までに行う
  • エ 切替えの後、動作確認が済んでから行う

正解: ウ — TTL短縮の変更自体が、既にキャッシュされた「TTL=86400の答え」が切れるまで世界のリゾルバに行き渡らない。だから旧TTL(86400秒=24時間)以上前に短縮しておく必要がある。イは「新TTLぶん前でよい」という誤解で、最も選ばれやすい誤答——効かせたいのは旧TTLの失効だ。ア・エでは切替時点で古い答えが最大24時間残ってしまう。

Q7. オープンリゾルバを放置してはならない理由として、最も適切なものはどれか。
  • ア ゾーン転送を要求され、ゾーン内の全レコードが流出するおそれがあるから
  • イ 送信元を詐称した問合せへの応答が第三者に送りつけられ、DDoS攻撃の踏み台にされるから
  • ウ 再帰問合せの処理でCPU使用率が上がり、権威応答が遅延するから
  • エ DNSSECの署名検証が省略され、偽の応答を受け入れてしまうから

正解: イ — オープンリゾルバの本質的な危険はDNSリフレクション攻撃の反射板・増幅器にされること。UDPゆえに送信元IPを詐称でき、小さな問合せが大きな応答となって攻撃対象に集中する。アはゾーン転送の制限不備という別の問題(オープンリゾルバの定義は再帰問合せの無制限受付け)。ウは性能の話で主因ではない。エはDNSSECと再帰開放に直接の関係がなく誤り。

Q8. DNSSECが提供する機能として、適切なものはどれか。
  • ア 問合せと応答を暗号化し、通信経路上での盗聴を防ぐ
  • イ ディジタル署名の検証により、応答が正当な管理者に由来し改ざんされていないことを確認する
  • ウ 権威サーバを冗長化し、名前解決の可用性を高める
  • エ 再帰問合せの送信元を認証し、オープンリゾルバ化を防ぐ

正解: イ — DNSSECは権威側の署名(RRSIG)をリゾルバが公開鍵で検証する仕組みで、提供するのは出自の認証と完全性。アの「暗号化(機密性)」は提供しない——最も出題される引っかけで、キャッシュポイズニング対策と盗聴対策の混同を突いてくる。ウの可用性はセカンダリ運用(ゾーン転送)の役割、エはアクセス制御の話でDNSSECの機能ではない。

Q9. DNSラウンドロビンによる負荷分散の問題点として、最も適切なものはどれか。
  • ア 複数のAレコードを1つの名前に登録すること自体が仕様違反である
  • イ サーバの死活を確認しないため、故障したサーバのアドレスも応答に含まれ続ける
  • ウ 応答のたびに権威サーバへの問合せが必須になり、権威サーバが過負荷になる
  • エ TTLを長くするほど分散の粒度が細かくなり、制御が難しくなる

正解: イ — DNSはサーバの状態を関知しないので、障害サーバへも利用者を案内し続ける。これがヘルスチェック付きの負荷分散装置やGSLBが必要になる理由だ。アは誤り——複数Aレコードは正当な設定。ウは逆で、キャッシュが効くぶん権威への問合せはむしろ減る。エも逆で、TTLが長いほどキャッシュに固定され分散は粗くなる。

Q10. digコマンドで社外向け公開名を問い合わせたところ、statusが「NXDOMAIN」であった。この結果から言えることとして、最も適切なものはどれか。
  • ア 問合せた名前がDNS上に存在しないという否定の応答が返った。レコードの登録漏れや名前の誤りを疑う
  • イ 名前解決の処理がサーバ側で失敗した。ゾーン転送切れやDNSSEC検証失敗を疑う
  • ウ 応答がキャッシュから返された。TTLが切れるまで待てば解決する
  • エ 権威サーバに到達できなかった。経路またはファイアウォールの問題を疑う

正解: ア — NXDOMAINは「その名前は存在しない」という権威側からの明確な否定であり、まず登録内容(レコードの有無・綴り)を確認するのが筋。イはSERVFAILの読み筋で、本問最大の対抗肢——「登録の問題(NXDOMAIN)」か「処理・経路の問題(SERVFAIL)」かの区別がこの問題の狙いだ。ウのキャッシュ応答自体はstatusでは分からない(TTLの減少で推測する)。エのような到達不能なら応答自体が返らない(タイムアウトする)ことが多い。
