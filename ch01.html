基礎講座 第1章

ネットワークの全体像とレイヤモデル

ネットワーク学習の最初の一歩は「層」という地図を手に入れること。この章ではOSI参照モデルとTCP/IPモデルという2枚の地図を手に入れ、URLを入力してからページが表示されるまでの通信の全行程を見渡す。以降の全章は、この地図の上に置かれていく。

学習目標:
  • OSI基本参照モデル7層とTCP/IPモデルの対応を、代表プロトコル・機器とセットで説明できる
  • カプセル化の流れと、Ethernet・IP・TCPヘッダの主要フィールドを挙げられる
  • 「L2の宛先は区間ごとに変わり、L3の宛先は端から端まで不変」の原則を構成図上で追える
  • ブラウザにURLを入力してからページが表示されるまでの通信を、順を追って説明できる
  • L2SW・L3SW・ルータ・FW・LB・プロキシが、それぞれどの層の情報を見て動くか即答できる

想定学習時間: 3〜4時間(例題・章末テスト込み)

1-1なぜ層に分けるのか — 「分業」という設計思想

まず、レイヤモデルが何の問題を解決するために生まれたかから考えよう。2台のコンピュータが通信するには、実に多くの仕事が必要になる。ケーブルに電気信号を流す、信号のかたまりに区切りを付ける、相手の住所を決めて届ける、途中で失われたデータを再送する、そしてWebページやメールといったアプリケーションの中身を解釈する——これらを全部ひとかたまりのソフトウェアで作ったらどうなるか。ケーブルを光ファイバに替えただけでWebブラウザを作り直す羽目になり、メーカーが違えば一切通信できない世界になる。

そこで通信の仕事を役割ごとの層(レイヤ)に分割し、「各層は自分の仕事だけをする」「各層はすぐ下の層のサービスを利用し、すぐ上の層にサービスを提供する」「層と層の間の受け渡し方(インタフェース)だけを決めておく」というルールを作った。これがレイヤモデルの設計思想だ。会社の分業と同じで、営業は製造の詳細を知らなくても仕事が回る。

層に分けたことで得られるものは3つある。第一に交換可能性。有線LANを無線LANに替えても、HTTPやTCPには一切手を入れなくてよい。下位層の実装を交換しても上位層は影響を受けない。第二に相互接続性。層ごとの取り決めが標準化されているから、メーカーが異なる機器同士でも通信できる。第三に切り分けやすさ。障害が起きたとき「ケーブルは生きているか(L1)→スイッチはフレームを運べているか(L2)→IPで相手に届くか(L3)→アプリは応答するか(L7)」と下から順に検証していける。

試験ではこう出る「なぜ層に分けるのか」が直接問われることは少ない。だが午後試験の障害切り分け問題は、「この症状はどの層の問題か」を特定する行為そのものだ。pingが通るのにWebが見えないならL3までは正常でL4以上を疑う——この発想は全章で使う。
合格の視点各層の役割は「下の層は、上の層が作った荷物の運び屋」と捉えると迷わない。運び屋は荷物(上位層のデータ)の中身を開けて見ない。この「中身に関知しない」という性質が、あとで学ぶカプセル化やトンネリングの理解につながる。

1-2OSI基本参照モデル — 7つの層の役割

層の分け方の代表が、ISOが策定したOSI基本参照モデルだ。通信の仕事を7つの層に整理したもので、現代のインターネットがOSIの設計そのままで動いているわけではないが、「通信を層で語るための共通言語」として試験でも実務でも使われ続けている。まず7層の全体像を眺めよう。

7
アプリケーション層人間の用事を通信サービスにする — HTTP, DNS, SMTP, DHCP, SNMP
6
プレゼンテーション層データの表現形式を整える — 文字コード変換、データ形式の取り決め
5
セション層会話(セション)の開始・維持・終了を管理する
4
トランスポート層アプリ同士の会話を端から端まで管理(ポート番号・信頼性) — TCP, UDP
3
ネットワーク層ネットワークをまたいで宛先まで届ける(IPアドレス・経路選択) — IP, ICMP
2
データリンク層同一リンク内の隣接機器間で運ぶ(MACアドレス) — イーサネット, 無線LAN
1
物理層ビットを電気・光・電波の信号に変えて流す — UTPケーブル, 光ファイバ

図1-1: OSI基本参照モデルの7層。下の層ほど「物理的な運搬」、上の層ほど「アプリケーションの意味」に近づく。

下から順に役割を一言で押さえる。物理層(L1)は0と1のビット列を電気・光・電波の信号に変換して伝送する層。ケーブルやコネクタの規格の世界だ。データリンク層(L2)は同じリンク(同一のLANセグメント)につながる隣接機器の間でデータのかたまり(フレーム)を運ぶ層。宛先の識別にはMACアドレスを使う。ネットワーク層(L3)は複数のネットワークをまたいで、最終的な宛先までパケットを届ける層。IPアドレスで宛先を指定し、ルータが経路を選択する。

トランスポート層(L4)は、届いた先のコンピュータの中で「どのアプリケーション宛か」をポート番号で識別し、必要なら再送や順序制御で通信の信頼性を保証する層。TCPとUDPがここに属する。セション層(L5)は会話の開始から終了までをひとまとまり(セション)として管理する層、プレゼンテーション層(L6)は文字コードやデータ形式など「表現」を揃える層、アプリケーション層(L7)はHTTPやメールなど、人間の用事そのものを通信サービスにする層だ。

正直に言うと、L5とL6は現代のプロトコルでは独立した実装を持たないことが多く、実務ではL7とまとめて「上位層」と扱われる。試験でもL5・L6の細かい機能を問う問題は多くない。重点はL1〜L4と、まとめてのL7に置いてよい。

試験ではこう出る午前IIでは「経路選択を行う層はどれか」「隣接ノード間の伝送制御を行う層はどれか」のように役割から層を答えさせる形が定番。「経路選択・IP=ネットワーク層」「隣接間・MAC=データリンク層」「ポート番号・再送=トランスポート層」の対応を反射で出せるようにしよう。

1-3TCP/IPモデルとの対応 — 試験の会話は「L2・L3・L4」で進む

OSIが「設計図としての理想形」だとすれば、実際のインターネットを作ってきたのはTCP/IPモデル(TCP/IPプロトコルスイート)だ。こちらは4層で整理される。両者の対応を表で確認しよう。

OSI参照モデルTCP/IPモデル代表プロトコル主な機器
アプリケーション層(7)
プレゼンテーション層(6)
セション層(5)		アプリケーション層HTTP, DNS, SMTP, DHCP, SNMP, TLSプロキシ, LB(L7), FW(上位機能)
トランスポート層(4)トランスポート層TCP, UDPFW, LB(L4)
ネットワーク層(3)インターネット層IP, ICMP, ARP(境界的)ルータ, L3スイッチ
データリンク層(2)
物理層(1)		ネットワークインタフェース層イーサネット, 無線LAN(IEEE 802.11), PPPL2スイッチ, ケーブル・NIC

TCP/IPモデルではOSIのL5〜L7をひとつのアプリケーション層に、L1〜L2をひとつのネットワークインタフェース層にまとめる。教材によってはL1とL2を分けて「5層モデル」と数えることもあり、本教材もその立場でL1(物理)・L2(データリンク)・L3(ネットワーク)・L4(トランスポート)・アプリケーション層の5階建てで話を進める。

大事なのは、試験・実務ともに「L2」「L3」「L4」という略称が共通言語だということ。「この2拠点はL2で延伸されている」「その制御はL4の情報で行う」といった言い回しが午後の問題文に当たり前に登場する。層の番号を聞いた瞬間に「見るアドレスは何か(MACか、IPか、ポートか)」が浮かぶようにしておきたい。

合格の視点7層すべてを均等に暗記する必要はない。合格に必要な解像度は「L1〜L4は深く、L5〜L6は名前と位置だけ、L7は各プロトコル(第6章で個別に)」という濃淡だ。学習時間もその配分で使おう。

1-4カプセル化 — データはヘッダで包まれて運ばれる

層のモデルが実際のデータの形にどう現れるか。それがカプセル化だ。送信側では、上位層のデータに各層が自分の管理情報(ヘッダ)を先頭に付け足しながら、下の層へ渡していく。荷物に伝票を貼り、それを箱に入れてまた伝票を貼る——という入れ子の構造になる。

アプリケーションデータ(例: HTTPリクエスト)
                     │
                     ▼ L4: TCPヘッダを付ける
          ┌─────────┬────────────────┐
          │TCPヘッダ │ データ          │        ←「セグメント」
          └─────────┴────────────────┘
                     ▼ L3: IPヘッダを付ける
     ┌────────┬─────────┬────────────────┐
     │IPヘッダ │TCPヘッダ │ データ          │   ←「パケット」
     └────────┴─────────┴────────────────┘
                     ▼ L2: Ethernetヘッダ+FCSを付ける
┌──────────────┬────────┬─────────┬──────────┬────┐
│Ethernetヘッダ │IPヘッダ │TCPヘッダ │ データ    │FCS │ ←「フレーム」
└──────────────┴────────┴─────────┴──────────┴────┘
                     ▼ L1: ビット列を電気・光・電波の信号として送出

図1-2: カプセル化。各層が先頭にヘッダを追加する(L2は末尾に誤り検出用のFCSも付ける)。受信側は逆の順に開封していく。

各層でのデータの呼び名(PDU: プロトコルデータユニット)も決まっている。L4で包んだものをセグメント(UDPではデータグラム)、L3で包んだものをパケット、L2で包んだものをフレーム、L1を流れるのはビットだ。受信側では逆に、L1から上がってきたフレームのヘッダを各層が外しながら(非カプセル化)、中身を上位層へ渡す。

それぞれのヘッダに何が書いてあるかを、主要フィールドだけ先取りしておこう。詳細はこの後の章で1つずつ扱うから、ここでは「どの層のヘッダに、どの情報が載るか」の対応を掴めばよい。

ヘッダ主なフィールド役割詳述
Ethernetヘッダ(L2)宛先MACアドレス / 送信元MACアドレス / タイプ(上位プロトコル識別)、末尾にFCS同一リンク内の隣接機器間の配送。FCSはフレームの誤り検出第2章
IPヘッダ(L3)送信元IPアドレス / 宛先IPアドレス / TTL / プロトコル番号 / フラグメント関連フィールド端から端までの配送。TTLはループ対策の寿命カウンタ第3章
TCPヘッダ(L4)送信元ポート番号 / 宛先ポート番号 / シーケンス番号 / ACK番号 / フラグ(SYN, ACK, FINなど)アプリの識別と、信頼性(再送・順序)の管理第5章
試験ではこう出る午前IIではPDUの名称(ネットワーク層=パケット、データリンク層=フレーム)がそのまま問われる。午後では一歩進んで「この装置はどのヘッダのどのフィールドを書き換えるか」が定番だ。例えばNATはIPヘッダのアドレスを、NAPTはさらにTCP/UDPヘッダのポート番号まで書き換える(第3章)。カプセル化の入れ子構造が頭にあれば、この種の問いは絵を描いて解ける。
例題 1-1PCのブラウザがWebサーバへデータを送信するとき、送信処理の順序として正しくなるよう、ア〜エを並べ替えよ。
ア IPヘッダを付加する イ TCPヘッダを付加する ウ Ethernetヘッダを付加しFCSを計算して付ける エ アプリケーションがHTTPリクエストのデータを生成する
解答と解説

エ → イ → ア → ウ

送信側は上の層から下の層へ、ヘッダを「外側に」追加しながら降りていく。アプリケーションデータ(エ)にL4のTCPヘッダ(イ)、L3のIPヘッダ(ア)、最後にL2のEthernetヘッダとFCS(ウ)。「送信は上から下、受信は下から上」と手順で覚えるより、図1-2の入れ子の絵を1回自分で描けば順序は間違えなくなる。

1-5大原則 — L2の宛先は区間ごとに変わり、L3の宛先は端から端まで不変

この教材で最初に叩き込みたい原則がこれだ。IPアドレス(L3)の送信元・宛先はエンドツーエンドで原則変わらないが、MACアドレス(L2)の送信元・宛先はルータを越える区間ごとに付け替えられる。フレームは「その区間の運び屋」、パケットは「最後まで届く荷物」だからだ。次の構成で確かめよう。

PC-A                      ルータR1                    ルータR2                  サーバB
IP : 192.168.1.10   LAN側IP: 192.168.1.1                              IP : 172.16.1.100
MAC: MAC-A          LAN側MAC: R1a                WAN側MAC: R2a        MAC: MAC-B
  │                 WAN側MAC: R1b                LAN側MAC: R2b             │
  │                       │                          │                    │
  └───── 区間1(社内LAN)────┘────── 区間2(WAN)─────────┘───── 区間3(サーバLAN)┘

図1-3: PC-AからサーバBへの通信路。ルータを境に3つのL2区間に分かれる。

PC-AからサーバBへパケットを送るとき、各区間を流れるフレーム・パケットのアドレスはこうなる。

区間送信元MAC宛先MAC送信元IP宛先IP
区間1: PC-A→R1MAC-AR1a(ルータR1)192.168.1.10172.16.1.100
区間2: R1→R2R1bR2a192.168.1.10172.16.1.100
区間3: R2→サーバBR2bMAC-B192.168.1.10172.16.1.100

ポイントは2つ。第一に、PC-Aは宛先サーバが自分と別のネットワークにいると判断すると(判断方法は第3章のサブネットの話)、フレームの宛先をデフォルトゲートウェイであるルータR1のMACにして送り出す。宛先IPはあくまで172.16.1.100のままで、である。第二に、各ルータは受け取ったフレームのL2ヘッダを剥がし、次の区間用のL2ヘッダを新しく付け直して転送する。このときIPヘッダのTTLを1減らす以外、送信元・宛先IPには手を触れない(NATを使う場合を除く——それは第3章で)。

なお、宛先IPに対応するMACアドレスを調べる仕組みがARPだ。「このIPの人、MAC教えて」というブロードキャストで、詳しい動作は第3章で扱う。ここでは「別セグメント宛のときにARPで解決するのは、宛先そのものではなくデフォルトゲートウェイのMAC」という結論だけ先に覚えてしまってよい。

試験ではこう出る午後の構成図問題で「図中の下線部のリンクを流れるフレームの宛先MACアドレスを答えよ」という形で頻出。「MACはその区間の次の受け取り手、IPは最終目的地」と唱えながら、区間の両端の機器を図から読み取れば確実に取れる設問だ。
例題 1-2図1-3の構成で、PC-AがサーバBへパケットを送信した。区間2(R1〜R2間)を流れるフレームについて、(1)宛先MACアドレス、(2)宛先IPアドレス、(3)送信元MACアドレスをそれぞれ答えよ。NATは使用していないものとする。
解答と解説

(1) R2a(ルータR2のWAN側MAC) (2) 172.16.1.100 (3) R1b(ルータR1のWAN側MAC)

区間2の「運び屋」はR1からR2への配送を担当するフレームなので、L2アドレスはその区間の両端インタフェースのもの(送信元R1b、宛先R2a)。一方L3の宛先はあくまで最終目的地のサーバB(172.16.1.100)、送信元も発信元のPC-A(192.168.1.10)のまま変わらない。「LAN側/WAN側のどちらのMACか」まで正確に答える癖を付けておくと午後で強い。

1-6機器と層の対応 — この箱はどの層まで見るか

ネットワーク機器は「受け取ったデータのどの層のヘッダまで見て転送・制御の判断をするか」で分類できる。午後の構成図には次の機器が記号で並ぶので、それぞれの守備範囲を一覧で頭に入れよう。

機器見る層判断材料役割の要点
リピータハブL1(信号のみ)受けた信号を全ポートへ再生・中継するだけ。現在はほぼ使われない
L2スイッチL2MACアドレスMACアドレステーブルを学習し、宛先MACのいるポートだけへフレームを転送(第2章)
ルータL3IPアドレスルーティングテーブルで経路を選択しネットワーク間を中継。ブロードキャストを堰き止める(第4章)
L3スイッチL2/L3MAC+IPルーティング機能を持つスイッチ。VLAN間ルーティングを専用ハードウェアで高速に行う(第2章)
ファイアウォール(FW)主にL3/L4IP+ポート+通信の状態通信を許可/遮断するセキュリティ装置。戻りの通信を自動許可するステートフルインスペクションが基本(第7章)
負荷分散装置(LB)L4/L7ポート、URLなど仮想IPで受けた要求を複数サーバへ振り分け、障害サーバを切り離す(第7章・第9章)

紛らわしいのはルータとL3スイッチの関係だ。どちらもL3で経路を選ぶが、L3スイッチは「多数のポートを持ち、社内のVLAN間中継を専用ハードウェアで高速処理する」のが得意で、ルータは「WAN回線への接続やNAT・VPNなど多機能な境界処理」が得意、という住み分けで理解しておく。社内の基幹にはL3スイッチ、インターネットや拠点間の出入り口にはルータ(またはFW)が置かれるのが典型構成だ。

また、FW・LB・プロキシは「転送する機器」ではなく通信を制御・仲介する機器という色が濃い。この3つは第6章・第7章で正面から扱うが、位置づけだけは今から構成図の中で意識しておこう。

合格の視点過去問の構成図を見るとき、機器名の横に「L2」「L3」「L4-7」と書き込む癖を付けよう。どの箱がどのヘッダを見るか・書き換えるかが見えると、「この通信はここで止まるはず」という推理が働くようになる。
例題 1-3次の(1)〜(3)の要件に最も適する機器を、下の語群から1つずつ選べ。
(1) 社内に多数あるVLAN間の通信を、高速に中継したい。
(2) 公開Webサーバ2台にクライアントからの要求を振り分け、片方の故障時は自動で切り離したい。
(3) 社内PCからのWebアクセスを代理で中継し、業務に関係ないサイトへのアクセスを制限したい。
語群: L2スイッチ / L3スイッチ / ルータ / 負荷分散装置 / プロキシサーバ
解答と解説

(1) L3スイッチ (2) 負荷分散装置 (3) プロキシサーバ

(1)はVLAN「間」の通信なのでL3の機能が必須。ルータでも可能だが「多数のVLAN」「高速に」とくればハードウェア転送のL3スイッチが定石。(2)は複数サーバへの振り分けと死活監視がキーワードで負荷分散装置。(3)は「代理」「アクセス制限」がキーワードでプロキシ。要件文の中の動詞(中継する・振り分ける・代理する)から機器を逆引きする練習が、午後の機器選定問題にそのまま効く。

1-7通信の一部始終 — URLを入力してからページが表示されるまで

ここまでの部品を全部つないで、通信の一部始終を追ってみよう。PCのブラウザに https://www.example.jp/ と入力してからページが表示されるまでに、裏では次の4幕の芝居が進行する。この流れは本教材全体の予告編だ。いまは各幕のあらすじが分かれば十分で、細部は挙げた章で必ず回収する。

ブラウザ(PC)                                        各サーバ
   │
   │ 第1幕: 名前解決(DNS)
   │   「www.example.jp のIPアドレスは?」──────▶ DNSサーバ(フルリゾルバ)
   │   ◀────────── 「203.0.113.80 です」            (UDP 53番)
   │
   │ 第2幕: コネクション確立(TCP)
   │   SYN ────────────────────────────────▶ Webサーバ 203.0.113.80:443
   │   ◀─────────────────────────── SYN+ACK
   │   ACK ────────────────────────────────▶  (3ウェイハンドシェイク完了)
   │
   │ 第3幕: 暗号化路の確立(TLS)
   │   ハンドシェイク: サーバ証明書の検証・暗号鍵の合意
   │
   │ 第4幕: ページの取得(HTTP)
   │   GET / ──────────────────────────────▶
   │   ◀──────────────── 200 OK(HTML本体)
   ▼  HTMLを解釈し、画像などを追加取得して描画

図1-4: URL入力からページ表示までの4幕。それぞれDNS・TCP・TLS・HTTPが主役を演じる。

第1幕: DNSによる名前解決。人間が扱うのはホスト名(www.example.jp)だが、L3の世界の住所はIPアドレスだ。PC内のスタブリゾルバはまず自分のキャッシュを見て、なければ設定されたDNSサーバ(フルリゾルバ)に問い合わせる。フルリゾルバは必要に応じてルートサーバ→jpのサーバ→example.jpの権威サーバとたどり、答えのIPアドレスを返す。この階層をたどる仕組みは第6章で詳述する。

第2幕: TCPコネクションの確立。宛先IPが分かったら、ブラウザはWebサーバのポート443(HTTPS)へTCPで接続する。SYN→SYN+ACK→ACKの3ウェイハンドシェイクで「これから会話を始めましょう」という合意を作る。TCPの信頼性の仕組みは第5章で詳述する。

第3幕: TLSによる暗号化。通信の盗聴や改ざん、サーバのなりすましを防ぐため、TLSハンドシェイクでサーバ証明書を検証し、暗号化に使う鍵を合意する。以降のHTTPのやり取りはすべて暗号化される。TLSと証明書は第7章で詳述する。

第4幕: HTTPでページを取得。確立した暗号化路の上で、ブラウザがGETリクエストを送り、サーバが200 OKとHTML本体を返す。ブラウザはHTMLを解釈し、画像やCSSを同様の手順で追加取得しながらページを描画する。HTTPの仕組みは第6章で詳述する。

そして、この4幕のすべてのパケットが、ここまでに学んだ土台の上を流れている。PCは各パケットを送るたびにデフォルトゲートウェイのMAC宛のフレームに載せ(ARP——第3章)、社内のL2スイッチがフレームを運び(第2章)、ルータがルーティングテーブルで経路を選び(第4章)、社内のプライベートアドレスはインターネットへ出る境界でNAPTにより変換される(第3章)。1つの「ページを見る」の裏に、全部の層の仕事が同時に走っているわけだ。

試験ではこう出る午後問題のシナリオは、たいていこのウォークスルーのどこか1〜2幕を切り取って「なぜ失敗するのか」「どう改善するか」を問う。名前解決はできるのに接続できない、証明書エラーが出る、特定拠点だけ遅い——症状がどの幕のものかを特定できれば、設問の半分は解けたようなものだ。
合格の視点この節の流れを、何も見ずに自分の言葉で3分間話せるか試してみてほしい。詰まった幕が、あなたがこれから重点的に学ぶべき章だ。学習が進むたびにこの節へ戻って、説明がどれだけ厚くなったかを確かめると成長が実感できる。
例題 1-4PCから https://www.example.jp/ へアクセスできない。調査したところ、(a) nslookupコマンドでwww.example.jpのIPアドレスは正常に得られる、(b) そのIPアドレスへのpingは応答がある、(c) ブラウザは「接続がタイムアウトしました」と表示する。この症状から、図1-4の4幕のうちどの段階に問題があると考えられるか。また、原因として考えられるものを1つ挙げよ。
解答と解説

段階: 第2幕(TCPコネクション確立)。原因の例: 経路上のファイアウォールがポート443宛のTCP通信を遮断している。

(a)でDNS(第1幕)は正常、(b)でL3の到達性も正常と切り分けられる。しかしブラウザが「タイムアウト」なので、TCPのSYNに対する応答が返っていない可能性が高い。pingは通るのにTCPが通らないときは、L4の情報(ポート番号)で遮断するFWやACLをまず疑う——層で切り分けるとはこういうことだ。この切り分けの型は午後試験でそのまま使える。

1-8標準化 — IETF・RFC・IEEEを知っておく

最後に、これから何度も登場する「規格の出どころ」を整理しておく。異なるメーカーの機器同士が会話できるのは、プロトコルが標準化されているからだ。誰がどの標準を作っているかを知っておくと、規格番号が固有名詞ではなく「住所」として読めるようになる。

IETF(Internet Engineering Task Force)はインターネットで使う技術の標準化を担う団体で、その成果文書がRFC(Request for Comments)だ。IP・TCP・HTTP・DNSなど、TCP/IP系のプロトコルの仕様はRFCとして番号付きで公開されている。試験対策としてRFC番号を暗記する必要はない。「TCP/IP系の仕様書=RFC、作っているのはIETF」という対応が分かれば十分だ。

IEEE(米国電気電子学会)は、LANの物理層・データリンク層の規格を「IEEE 802委員会」で標準化している。ここは番号が試験の選択肢に直接出るので、グループの見分け方を覚えておこう。802.3系が有線イーサネット、802.11系が無線LAN、802.1系がブリッジングや管理の横断的な規格(VLANの802.1Q、認証の802.1Xなど)だ。第2章で出てくる802.1Q(VLANタグ)や802.3ad(リンクアグリゲーション)も、この住所体系の中の一員である。

団体担当領域成果物・代表規格
IETFインターネットのプロトコル(主にL3以上)RFC(IP, TCP, HTTP, DNSなど)
IEEE 802委員会LANの物理層・データリンク層802.3(イーサネット), 802.11(無線LAN), 802.1Q(VLAN), 802.1X(認証), 802.3ad(リンクアグリゲーション)
合格の視点「802.3=有線、802.11=無線、802.1=橋渡し・管理」と3グループで覚えれば、初見の規格番号でもおおよその守備範囲を推測できる。番号の丸暗記が必要になったら、その規格を学ぶ章(第2章など)で中身とセットで覚えるのが早い。

チェックテスト(10問)

全問正解できたら次章へ。迷った問題は該当セクションに戻ること。

Q1. OSI基本参照モデルにおいて、ポート番号によってアプリケーションを識別し、エンドツーエンドの信頼性のあるデータ転送を担う層はどれか。
  • ア セション層
  • イ トランスポート層
  • ウ ネットワーク層
  • エ データリンク層

正解: イ — ポート番号・再送・順序制御とくればトランスポート層(L4)で、代表はTCP(1-2節)。ウのネットワーク層は「端から端まで届ける」層だが、識別はIPアドレスで行い信頼性の保証はしない。アのセション層は会話の開始・終了の管理で、ポート番号は扱わない。

Q2. ネットワーク層のPDU(プロトコルデータユニット)の名称はどれか。
  • ア セグメント
  • イ フレーム
  • ウ パケット
  • エ データグラム

正解: ウ — L3=パケット、L2=フレーム、L4(TCP)=セグメント(1-4節)。エのデータグラムはUDPのPDUの呼び名で、トランスポート層のもの。層とPDU名の対応は午前IIの即答問題なので確実に。

Q3. PCがWebサーバへデータを送信するときのカプセル化の順序として正しいものはどれか。
  • ア Ethernetヘッダ → IPヘッダ → TCPヘッダの順に付加する
  • イ TCPヘッダ → IPヘッダ → Ethernetヘッダの順に付加する
  • ウ IPヘッダ → TCPヘッダ → Ethernetヘッダの順に付加する
  • エ TCPヘッダ → Ethernetヘッダ → IPヘッダの順に付加する

正解: イ — 送信側は上位層から順に、L4(TCP)→L3(IP)→L2(Ethernet)とヘッダを外側へ重ねていく(1-4節)。アは受信側で「外す」順序なら正しいが、付加する順序としては逆。フレームの先頭からの並び(Ethernet→IP→TCP)と、付加する時間的順序(TCP→IP→Ethernet)を混同しないこと。

Q4. ルータを1台通過するたびに値が1ずつ減らされる、IPヘッダのフィールドはどれか。
  • ア TTL
  • イ シーケンス番号
  • ウ FCS
  • エ プロトコル番号

正解: ア — TTLは経路ループでパケットが永遠に回り続けるのを防ぐ寿命カウンタで、0になると破棄される(1-4節、詳細は第3章)。イのシーケンス番号はTCPヘッダ、ウのFCSはEthernetフレームの誤り検出フィールドで、そもそもIPヘッダのものではない。エのプロトコル番号は上位プロトコルの識別子で通過中に変化しない。

Q5. PC(192.168.1.10)が別セグメントのサーバ(172.16.1.100)へパケットを送信するとき、PCが送出するフレームの宛先MACアドレスはどれか。
  • ア サーバのMACアドレス
  • イ デフォルトゲートウェイのMACアドレス
  • ウ ブロードキャストMACアドレス
  • エ 経路上の最終ルータのMACアドレス

正解: イ — 宛先が別セグメントの場合、L2の宛先は「次の受け取り手」であるデフォルトゲートウェイになる(1-5節)。アは宛先IPと宛先MACを混同した典型的な誤り。L2アドレスがサーバ本人のMACになるのは最後の区間だけだ。ウのブロードキャストはARP要求のときに使うもので、データ本体のフレームの宛先ではない。

Q6. 図1-3の構成(PC-A—R1—R2—サーバB、NATなし)で、区間2(R1〜R2間)を流れるパケットの送信元IPアドレスはどれか。
  • ア PC-AのIPアドレス
  • イ R1のWAN側IPアドレス
  • ウ R2のWAN側IPアドレス
  • エ サーバBのIPアドレス

正解: ア — NATを使わない限り、送信元・宛先IPはエンドツーエンドで不変(1-5節)。イは「MACなら正しい」引っかけで、区間2の送信元MACは確かにR1のWAN側だが、IPは書き換えられない。この「L2は替わる・L3は替わらない」の対比が本章最大の原則だ。

Q7. HTTPSのWebページを表示するとき、通信が行われる順序として正しいものはどれか。
  • ア TCP接続 → DNS名前解決 → TLSハンドシェイク → HTTPリクエスト
  • イ DNS名前解決 → TLSハンドシェイク → TCP接続 → HTTPリクエスト
  • ウ DNS名前解決 → TCP接続 → TLSハンドシェイク → HTTPリクエスト
  • エ DNS名前解決 → TCP接続 → HTTPリクエスト → TLSハンドシェイク

正解: ウ — 宛先IPが分からなければ接続できないのでDNSが最初、TLSはTCPコネクションの上で動くのでTCPの後、HTTPは暗号化路ができてから(1-7節)。イはTLSがTCPより先になっている点が誤り。TLSのハンドシェイクパケット自体がTCPで運ばれることを思い出せば順序は導ける。

Q8. L3スイッチの説明として、最も適切なものはどれか。
  • ア MACアドレステーブルだけを用いてフレームを転送する装置
  • イ ルーティング機能を持ち、VLAN間の通信を高速に中継できるスイッチ
  • ウ URLの内容に基づいてWebアクセスを代理・制御する装置
  • エ 仮想IPアドレス宛の要求を複数のサーバに振り分ける装置

正解: イ — L3スイッチ=ルーティングもできるスイッチで、社内のVLAN間中継が主戦場(1-6節)。アはL2スイッチの説明で、L3スイッチとの違いは「IPを見て経路選択できるか」にある。ウはプロキシ、エは負荷分散装置。機器名と「見る層・判断材料」の対応表(1-6節)を思い出そう。

Q9. 有線イーサネットの規格を標準化している団体・規格群はどれか。
  • ア IETFのRFC
  • イ IEEE 802.3
  • ウ IEEE 802.11
  • エ ISOのOSI基本参照モデル

正解: イ — 有線イーサネットはIEEE 802.3系(1-8節)。ウの802.11は無線LANで、最も紛らわしい。アのRFCはTCP/IP系プロトコル(L3以上が中心)の仕様書。エのOSI参照モデルは「層の考え方」の枠組みであって、イーサネットという具体的な規格を定めたものではない。

Q10. pingでは宛先サーバから応答があるのに、そのサーバのWebページ(TCPポート443)には接続できない。最初に疑うべき箇所として最も適切なものはどれか。
  • ア LANケーブルの断線
  • イ DNSサーバの障害
  • ウ 経路上のファイアウォールによるポート443の遮断
  • エ デフォルトゲートウェイの設定誤り

正解: ウ — pingが通る時点でL1〜L3は正常と切り分けられる(1-7節、例題1-4)。残るのはL4以上、つまり特定ポートの通信だけを止めている装置=FWがまず疑わしい。ア・エはL1〜L3の障害なのでpingも失敗するはず。イはIPアドレス指定の接続確認ができている前提なら除外できる(名前解決の問題ならpingの宛先解決も失敗する)。
