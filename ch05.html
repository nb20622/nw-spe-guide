基礎講座 第5章

TCPとUDP — トランスポート層

IPは「届くよう努力する」だけで、届いた保証はしてくれない。その上で信頼性を作り込むTCPと、あえて作り込まないUDP。この章を終えると、スループットの理論値計算と「なぜ長距離回線は速度が出ないのか」「なぜICMPを全部遮断すると通信が壊れるのか」に答えられるようになる。

学習目標:
  • ポート番号と5タプルでコネクションが識別される仕組みを説明できる
  • 3ウェイハンドシェイクと切断シーケンスを、シーケンス番号・ACK番号込みで書ける
  • ウィンドウサイズとRTTからスループットの理論値を計算できる
  • MTUからMSSを計算し、PMTUDがICMP遮断で壊れる仕組みを説明できる
  • TCPとUDPの使い分け、QUICの位置づけを説明できる

想定学習時間: 4〜5時間(例題・章末テスト込み)

5-1ポート番号と5タプル — 通信の「宛先の中の宛先」

IPアドレスはホスト(のインタフェース)を特定するが、そのホストの中ではWebサーバもメールサーバも同時に動いている。届いたデータをどのアプリケーションに渡すべきか——これを解決するのがポート番号(16ビット、0〜65535)だ。トランスポート層の仕事は、大づかみに言えば「IPが運んできたデータを正しいアプリケーションに振り分けること」と「(TCPの場合)その通信に信頼性を与えること」の2つである。

1本の通信(コネクション)は、プロトコル(TCP/UDP)・送信元IP・送信元ポート・宛先IP・宛先ポートの5つの値の組=5タプルで一意に識別される。同じPCから同じWebサーバへブラウザのタブを2枚開いても、送信元ポートが異なるので別コネクションとして区別できる。この5タプルは、ファイアウォールのポリシー(第8章)やNAPTの変換テーブルなど、この先あらゆる場面で「通信を識別する単位」として顔を出す。

ポート番号は範囲ごとに役割が決まっている。ウェルノウンポート(0〜1023)はサーバ側の代表的なサービスに予約された番号。1024〜49151は登録済みポート、49152〜65535はクライアントが一時的に使うエフェメラルポート(短命ポート)だ。クライアントはOSが自動で選んだエフェメラルポートを送信元にして、サーバのウェルノウンポート宛に接続する、というのが基本の形になる。

5-2TCPヘッダの主要フィールド

TCP(Transmission Control Protocol)は、IPという「届くかどうか分からない配送網」の上に、順序どおり・欠けなく・相手の処理能力と回線の混雑に配慮してデータを届ける仕組みを作り込んだプロトコルだ。その道具立てはすべてヘッダのフィールドに現れているので、まずヘッダの地図を頭に入れる。TCPヘッダはオプションなしで20バイトある。

0                   15 16                  31 (ビット)
┌──────────────────────┬──────────────────────┐
│  送信元ポート (16)     │  宛先ポート (16)      │
├──────────────────────┴──────────────────────┤
│        シーケンス番号 (32)                    │
├─────────────────────────────────────────────┤
│        確認応答番号 = ACK番号 (32)            │
├────────┬─────────────┬──────────────────────┤
│オフセット│ フラグ       │  ウィンドウサイズ (16) │
│  (4)   │ SYN/ACK/FIN…│                      │
├────────┴─────────────┼──────────────────────┤
│  チェックサム (16)     │  緊急ポインタ (16)    │
├──────────────────────┴──────────────────────┤
│  オプション (MSS、ウィンドウスケール、SACK許可 …)│
└─────────────────────────────────────────────┘

図5-1: TCPヘッダの構造(基本部20バイト)。この章の主役はシーケンス番号・ACK番号・フラグ・ウィンドウサイズの4つだ。

フィールドの役割をまとめる。

  • シーケンス番号(32ビット) — このセグメントが運ぶデータの先頭バイトが、バイト列全体の何番目かを示す。TCPはデータを「番号付きのバイトの流れ」として扱う。
  • 確認応答番号(ACK番号) — 「ここまで受け取った。次はこの番号のバイトをください」という受領通知。累積確認方式で、この番号の直前までの全バイトの受領を意味する。
  • フラグ — セグメントの性格を示すビット群。SYN(接続開始・シーケンス番号の同期)、ACK(確認応答番号が有効)、FIN(送るデータはもうない=切断開始)、RST(強制切断・接続拒否。閉じたポートへの接続要求にはRSTが返る)、PSH(たまったデータを即座にアプリへ渡す)、URG(緊急データ)。
  • ウィンドウサイズ(16ビット) — 「私はいまACKを待たずにあと何バイト受け取れるか」を相手に伝える値。フロー制御(5-5)の要。16ビットなので素の最大値は65,535バイト——この上限が後でスループットの話(5-7)に効いてくる。
  • オプション — MSS(5-8)、ウィンドウスケール(ウィンドウ値を2のn乗倍に拡大)、SACK許可(5-4)などを接続開始時に取り決める。
試験ではこう出る午前IIでは「TCPヘッダにあってUDPヘッダにないフィールドはどれか」(→シーケンス番号、ACK番号、ウィンドウサイズなど)、「RSTフラグの意味」などが問われる。UDPヘッダは送信元ポート・宛先ポート・長さ・チェックサムの8バイトだけ——この対比で覚えると効率がいい。

5-33ウェイハンドシェイクとコネクション切断

TCPは通信の前に、互いに「これから使うシーケンス番号の初期値」を教え合い、送受信の準備を確認する。これが3ウェイハンドシェイクだ。手順は3往復ならぬ3手。

  1. クライアント→サーバ: SYN(シーケンス番号の初期値x を提示)
  2. サーバ→クライアント: SYN/ACK(自分の初期値y を提示しつつ、ACK番号 = x+1 で「xを受け取った、次はx+1から」と応答)
  3. クライアント→サーバ: ACK(ACK番号 = y+1)。これで両者ESTABLISHED(確立)状態。

SYNはデータを運ばないがシーケンス番号を1つ消費する。だからACK番号が「x+1」になる——この+1は穴埋め問題の定番だ。

切断は開始よりも丁寧で、4手かかる。TCPは全二重(双方向それぞれ独立のバイトの流れ)なので、「私はもう送るものがない」を双方が別々に宣言する必要があるからだ。先に切断を言い出した側(アクティブクローズ側)がFINを送り、相手のACK、相手のFIN、自分のACK、で完了する(相手のACKとFINがまとまって3手になることもある)。

クライアント                          サーバ
    │ ── SYN (seq=x) ──────────────→ │  接続要求
    │ ←────────── SYN/ACK (seq=y,   │
    │              ack=x+1) ──       │
    │ ── ACK (ack=y+1) ────────────→ │  確立(ESTABLISHED)
    │ ⇄  データ転送 ⇄               │
    │ ── FIN/ACK ──────────────────→ │  切断開始(アクティブクローズ)
    │ ←──────────────────── ACK ──  │
    │ ←──────────────── FIN/ACK ──  │  サーバ側も送信終了
    │ ── ACK ──────────────────────→ │
 [TIME_WAIT]                          │  ← 先に切った側が2MSL待つ
    │ (2MSL経過後 CLOSED)            │

図5-2: 接続確立(3手)と切断(4手)のシーケンス。TIME_WAITに入るのは「先に切断を言い出した側」である点に注意。

最後のACKを送ったアクティブクローズ側は、すぐにはコネクションを消さず、TIME_WAIT状態で一定時間(セグメントの最大生存時間MSLの2倍=2MSL)待つ。理由は2つあり、これは記述問題の頻出ネタだ。

  • 最後のACKが失われたときの保険 — ACKが届かなければ相手はFINを再送してくる。コネクションを消してしまっていると応答できない(RSTを返して相手を混乱させる)。
  • 古いセグメントの排除 — 同じ5タプルのコネクションをすぐ再利用すると、ネットワーク内をさまよっていた旧コネクションのセグメントが新コネクションに紛れ込むおそれがある。2MSL待てば旧セグメントは死に絶える。
試験ではこう出る「TIME_WAITはどちら側に発生するか」「なぜ待つのか」は午前II・午後の両方で出る。先にFINを送った側(アクティブクローズ側)が、最終ACKの喪失に備え、かつ旧セグメントの混入を防ぐために2MSL待つ——ここまで一息で言えるように。Webサーバのように大量の短い接続を自分から切る側では、TIME_WAITのコネクションが多数残るのは正常な動作だ、という点も押さえておこう。

5-4信頼性の仕組み — シーケンス番号・ACK・再送

IPネットワークではパケットの喪失・重複・順序逆転が普通に起きる。TCPはこれをシーケンス番号とACKの2つだけで立て直す。受信側は、届いたセグメントのシーケンス番号を見てバイト列を正しい順に並べ直し、重複は捨てる。そして「次に欲しいバイト番号」をACKで返す。送信側は、送ったデータのACKが返らなければ再送する。信頼性の正体はこの「番号を振る・受領を確認する・だめなら送り直す」の3点セットだ。

再送のきっかけは2種類ある。

  • 再送タイマ(RTO)の満了 — セグメント送信時にタイマを起動し、ACKが来ないままタイムアウトしたら再送する。RTO(Retransmission Time Out)はRTT(往復遅延時間)の実測から動的に計算される。確実だが、タイマ満了を待つぶん回復が遅い。
  • 重複ACKによる高速再転送 — 受信側は歯抜けのセグメント(例: 1〜1000は届いたが1001〜2000が抜け、2001〜が届いた)を受け取るたびに、「次は1001が欲しい」という同じACKを繰り返し返す。送信側は同一ACKを3回重複して受け取ったら、タイマを待たずに該当セグメントを即再送する。これが高速再転送(fast retransmit)で、タイムアウトを待つより格段に速く回復できる。

累積ACKには「歯抜けの後ろがどこまで届いているか伝えられない」という弱点がある。これを補うのがSACK(選択確認応答)オプションで、「1001〜2000は未着だが、2001〜4000は受領済み」のように飛び飛びの受領範囲を伝えられる。送信側は本当に失われた部分だけを再送すればよくなる。接続開始時にSACK許可オプションで合意して使う、という程度に押さえておけば試験には足りる。

例題 5-13ウェイハンドシェイク完了直後、クライアントの次のシーケンス番号は1001、サーバの次のシーケンス番号は3001である。次の順でデータをやり取りするとき、空欄a〜cを埋めよ。
(1) クライアント→サーバ: seq=1001 で500バイトのデータを送信
(2) サーバ→クライアント: ACK番号=［ a ］の確認応答と同時に、seq=3001 で1,000バイトのデータを送信
(3) クライアント→サーバ: ACK番号=［ b ］の確認応答
(4) クライアント→サーバ: seq=［ c ］で次のデータを送信
解答と解説

a=1501、b=4001、c=1501

(1)でクライアントはバイト1001〜1500(500バイト)を送った。サーバのACK番号は「次に欲しいバイト」なので a=1501。(2)でサーバはバイト3001〜4000(1,000バイト)を送ったから、クライアントのACK番号は b=4001。クライアント自身の送信バイト列は1500まで使ったので、次のセグメントは c=1501 から始まる。

ポイントは2つ。ACK番号=受信した最終バイト+1(=次に欲しいバイト)であること、そして双方向のシーケンス番号は独立で、自分の送信番号は相手から受けたデータでは進まないこと。午後の穴埋めでは双方向が入り交じった表で出るので、クライアント側・サーバ側の番号を別々の行にメモして追うと混乱しない。

5-5ウィンドウ制御とフロー制御

「1セグメント送るたびにACKを待つ」方式では、回線がどれだけ太くてもRTTごとに1セグメントしか送れず、遅すぎて話にならない。そこでTCPは、ACKを待たずに連続送信してよいデータ量=ウィンドウを設け、その範囲内のセグメントをまとめて送り出す。ACKが返ってくるたびにウィンドウが前に滑っていく様子からスライディングウィンドウと呼ばれる。回線を待ち時間で遊ばせない、これがウィンドウ制御の本質だ。

ではウィンドウの大きさは誰が決めるのか。ひとつの決め手は受信側の都合である。受信側は届いたデータをいったん受信バッファにため、アプリケーションが読み出すのを待つ。アプリの処理が遅ければバッファは埋まっていく。そこで受信側は、ACKを返すたびにヘッダのウィンドウサイズフィールドで「バッファの空きはあと何バイト」と申告する。送信側は未確認のまま送っていい量をこの値以下に抑える。これがフロー制御——受信側の処理能力に合わせて送信をブレーキする仕組みだ。

バッファが満杯になると受信側はウィンドウサイズ0を通知し、送信側は送信を停止する(ゼロウィンドウ)。その後、送信側は小さなウィンドウプローブを時々送って空きの回復を確認する。「ウィンドウ更新の通知が失われても、プローブがあるのでデッドロックしない」という設計だ。

合格の視点フロー制御と次節の輻輳制御は混同しやすいが、守っている相手が違う。フロー制御は受信側(のバッファ)を守る。輻輳制御はネットワーク(途中の回線・ルータ)を守る。前者は相手が申告してくれる値(rwnd)、後者は送信側が自分で推測する値(cwnd)。実際の送信可能量は両者の小さいほうで決まる。この一段落が言えれば、この2つに関する午前II問題はほぼ落とさない。

5-6輻輳制御 — スロースタートから輻輳回避へ

受信側に余裕があっても、途中のネットワークが混雑(輻輳)していれば、送った先からパケットが捨てられる。捨てられたら再送する——だが全員が再送すればさらに混雑し、再送がまた捨てられる。放っておくとネットワーク全体が輻輳の連鎖で崩壊する。これを防ぐため、TCPの送信側は輻輳ウィンドウ(cwnd)という自主規制値を持ち、ネットワークの混み具合を推測しながら送信量を調節する。混み具合を直接教えてくれる装置はないので、パケット喪失を輻輳のサインとみなすのが基本戦略だ。

制御は2つのフェーズを行き来する。

  • スロースタート — 接続直後はネットワークの容量が全く分からないので、cwndを小さな値から始める。ACKが返ってくるたびにcwndを増やし、結果としてRTTごとに倍々で送信量が増えていく。「スロー」という名前だが増え方は指数的で、出発点が控えめなことを指した命名だ。
  • 輻輳回避 — cwndがしきい値(ssthresh)に達したら、増加をRTTごとに1セグメントぶんずつの直線的な増加に切り替え、限界を慎重に探る。

喪失を検知したときの反応は検知方法で異なる。タイムアウト(RTO)による検知は重度の輻輳とみなし、ssthreshをそのときのウィンドウの半分に設定した上でcwndを初期値まで戻し、スロースタートからやり直す。一方、重複ACKによる高速再転送で検知したときは「ACKが返ってきている=ネットワークはまだ流れている」ので、cwndを半分に減らす程度にとどめて輻輳回避を続ける(高速リカバリ)。増やすときは慎重に、減らすときは大胆に——この非対称が、皆で回線を分け合うための作法になっている。

cwnd
  │                広告ウィンドウ(rwnd)上限 ─────────
  │            ／│
  │   ssthresh ／ │←喪失検知: ssthresh=当時の半分
  │ ---------／---│---------            ／
  │        ／     │      新ssthresh ／ ← 以降は直線増加
  │      ／       │       ------／
  │    ／指数増加  │         ／
  │  ／(スロー     │       ／ 直線増加(輻輳回避)
  │／  スタート)   ↓     ／
  └──────────────────────────────────────→ 時間

図5-3: cwndの typical な変化。指数的に立ち上がり(スロースタート)、しきい値からは直線的に探り(輻輳回避)、喪失で減らして再び探る。

試験ではこう出る午前IIでは「スロースタートの説明として適切なものはどれか」が定番。「ACKを受けるたびに輻輳ウィンドウを増やし、RTTごとに指数的に送信量を増やす。しきい値到達後は直線的な増加(輻輳回避)に移る」という流れと、フロー制御(受信側保護)との区別が問われる。cwndの初期値やアルゴリズムの品種(Renoなど)の暗記までは不要だ。

5-7スループットの理論値 — ウィンドウサイズ÷RTT

ウィンドウ制御を理解すると、TCPの速度の限界が計算できるようになる。送信側はウィンドウサイズWぶんのデータを送ると、ACKが返るまで次を送れない。ACKが返るまでの時間はRTT(往復遅延時間)。つまりどれだけ回線が太くても、RTTあたりWバイトしか送れない。

スループットの上限 = ウィンドウサイズ ÷ RTT

この式は回線帯域がどこにも出てこないのが肝だ。回線が1Gビット/秒あっても、ウィンドウとRTTが作る天井のほうが低ければ、速度はそこで頭打ちになる。東京〜海外拠点のようにRTTが大きい回線(ロングファットパイプ)で「帯域はあるのにファイル転送が遅い」という古典的な症状の正体がこれで、ヘッダのウィンドウフィールドが16ビット(最大65,535バイト)しかないことが根本原因になる。対策がウィンドウスケールオプション——接続時に「ウィンドウ値を2のn乗倍で解釈する」と合意して、実効ウィンドウを拡大する。加えて5-6の輻輳制御による天井(cwnd)もあるため、実際のスループットはmin(rwnd, cwnd)÷RTTが上限となる。

例題 5-2RTTが100ミリ秒の回線で、TCPのウィンドウサイズが64kバイト(64×103バイトとする)のとき、(1) スループットの理論上の最大値は何Mビット/秒か。(2) この通信を1Gビット/秒の回線で行った場合、回線帯域を使い切れるか。(3) 1Gビット/秒・RTT 100ミリ秒を使い切るのに必要なウィンドウサイズは何Mバイトか。
解答と解説

(1) 5.12Mビット/秒 (2) 使い切れない (3) 12.5Mバイト

(1) 100ミリ秒ごとに64kバイト送れるのだから、
64×103バイト × 8ビット = 512×103ビット を 0.1秒で送る → 512×103 ÷ 0.1 = 5.12×106ビット/秒 = 5.12Mビット/秒

(2) 理論上限が5.12Mビット/秒なので、1Gビット/秒の回線でも0.5%程度しか使えない。回線を増速しても速くならない——原因はウィンドウとRTTだからだ。

(3) 必要ウィンドウ = 帯域×RTT(帯域遅延積)= 109ビット/秒 × 0.1秒 = 108ビット = 12.5×106バイト = 12.5Mバイト。標準の65,535バイトでは全く足りず、ウィンドウスケールオプションが必須と分かる。

計算のコツは「バイト→ビットの×8」と「ミリ秒→秒の÷1000」を最初に片付けること。単位変換のミスがこの種の問題の失点原因のほぼ全てだ。

試験ではこう出る「ウィンドウサイズとRTTからスループットを計算せよ」は午前II・午後を通じた頻出計算問題。逆算パターン(目標スループットに必要なウィンドウサイズ=帯域遅延積を求めさせる)も出る。式は1本しかないので、単位変換込みで3回手を動かせば確実に取れる得点源になる。

5-8MTU・MSSとフラグメンテーション、PMTUD

データリンクには一度に運べるサイズの上限=MTU(Maximum Transmission Unit)がある。イーサネットなら1500バイト——これはIPヘッダを含むIPパケット全体のサイズ上限だ(第2章)。一方TCPにはMSS(Maximum Segment Size)という値があり、こちらはTCPのデータ部分(ペイロード)だけの上限を指す。関係は単純な引き算で、

MSS = MTU − IPヘッダ(20バイト) − TCPヘッダ(20バイト)

イーサネットならMSS = 1500 − 40 = 1460バイト。MSSは3ウェイハンドシェイクのSYNに載せて互いに申告し、小さいほうに合わせる。「ヘッダ込みか、データだけか」——MTUとMSSの違いはこの一点に尽きる。

ではMTUの異なる経路をまたぐとどうなるか。MTU 1500で送り出したパケットが、途中のMTU 1400の区間(トンネルやVPNで発生しがち)に差しかかると、そのままでは運べない。IPv4のルータはパケットを分割(フラグメンテーション)して通すことができ、分割された断片は最終的な宛先ホストで再組立てされる(途中のルータは組み立てない)。ただし分割はルータの負荷になり、断片が1つでも落ちると全体を再送する羽目になるため、できれば避けたい。なおIPv6では途中のルータによる分割は仕様上行われない(第11章)。

そこで現代のホストは、最初から経路上の最小MTUを見つけて、それに収まるサイズで送る戦略を取る。これが経路MTU探索(PMTUD: Path MTU Discovery)だ。仕組みは巧妙で、パケットのIPヘッダにDFビット(Don't Fragment: 分割禁止)を立てて送る。MTUを超える区間に差しかかったルータは、分割が禁止されているので転送できず、パケットを破棄して送信元にICMPエラー(タイプ3・コード4「分割が必要だがDF指定」)を返す。このエラーには通過できるMTU値が含まれており、送信元はサイズを下げて再送する。これを繰り返して経路の最小MTUに収束する。

[送信ホスト]──MTU1500──[R1]──MTU1400(VPN区間)──[R2]──MTU1500──[サーバ]
     │                    │
     │ ①DF付き1500バイト → │ 転送不可(分割禁止のため)
     │ ← ②ICMP タイプ3コード4「MTU1400まで」
     │ ③1400バイトに下げて再送 →  →  →  →  →  ✓届く

  ★もし途中のFWがICMPを全て遮断していると…
     ②が送信ホストに届かない
     → ホストは1500バイトで送り続け、静かに破棄され続ける
     → 小さいパケット(ハンドシェイクや小さなページ)だけ通る
       「接続はできるのに大きなデータで固まる」ブラックホール発生

図5-4: PMTUDの動作と、ICMP遮断による経路MTUブラックホール。午後問題の定番シナリオだ。

ここに午後試験の頻出トラブルが潜む。セキュリティ目的でファイアウォールがICMPを一律遮断していると、②のICMPエラーが送信元に届かない。送信元は破棄されたことを知るすべがなく、大きなパケットを送っては失われ続ける。症状は特徴的で、「TCP接続は確立する(SYNは小さいので通る)のに、大きなデータの転送だけが進まず固まる」。対策は、(1) ファイアウォールでICMPタイプ3コード4(到達不能・分割必要)は許可する、(2) 経路上のルータでSYNのMSS値を書き換えて最初から小さく送らせる(MSSクランプ)、のどちらかが定番解答だ。

例題 5-3(1) MTUが1500バイトのイーサネット区間で、TCPのMSSは何バイトか。(2) PPPoE(ヘッダ8バイト)を使うアクセス回線ではMTUが1492バイトになる。このときのMSSは何バイトか。(3) この回線のルータが「MSSクランプ」を行う理由を、PMTUDとの関係で述べよ。
解答と解説

(1) 1460バイト (2) 1452バイト

(1) 1500 − IP 20 − TCP 20 = 1460。(2) PPPoEでMTUが1492に縮むので、1492 − 40 = 1452。引き算そのものより「MSSはTCPデータ部だけ」という定義を思い出せるかが勝負だ。

(3) 経路の途中にMTUの小さい区間があることをエンドホストは知らないため、本来はPMTUD(ICMPエラー)でサイズを学習する必要がある。しかしICMPが途中で遮断されているとPMTUDが機能せず、大きなセグメントが破棄され続ける通信不能が起きる。ルータがハンドシェイク時のMSS値を回線のMTUに合わせて小さく書き換えておけば、ICMPに頼らずに最初から通るサイズで送信させられるから。ICMP遮断環境でも壊れない予防策として定番であり、記述問題でこのまま使える論理だ。

5-9UDPの特性と用途、そしてQUIC

UDP(User Datagram Protocol)は、TCPが作り込んだものをすべて省いたプロトコルだ。ヘッダは送信元ポート・宛先ポート・長さ・チェックサムの8バイトだけ。コネクション確立なし、再送なし、順序保証なし、フロー制御・輻輳制御なし。「ポートによる振り分け」と「化けの検出(チェックサム)」だけを提供する。

手抜きに見えるが、これは意図的な設計だ。UDPが選ばれる理由は用途を見ると腑に落ちる。

  • DNS(53)・NTP(123)・SNMP(161/162)・DHCP(67/68) — 「小さな問合せに小さな応答が1回」の通信。3ウェイハンドシェイクを張るコスト(1RTT+状態管理)のほうがもったいない。応答がなければアプリ自身が再問合せすれば済む。DNSの詳細は第6章で扱う。
  • 音声・映像のリアルタイム転送(RTP) — 0.1秒前の音声を再送してもらっても、再生時刻はもう過ぎている。再送で遅れるくらいなら欠けたまま進むほうがよい用途では、TCPの信頼性はむしろ邪魔になる。
  • ブロードキャスト/マルチキャスト — 1対多の送信は、1対1の接続を前提とするTCPでは原理的にできない。DHCPの発見パケットなどはUDPでなければ成立しない。

そして近年、UDPの上にアプリケーション層で信頼性を作り直すという第三の道が主流になりつつある。QUICだ。HTTPをTCP+TLSで運ぶ従来方式には、接続確立とTLSハンドシェイクで複数RTTかかる、1つのパケット喪失がTCPコネクション上の全データを止める(HoLブロッキング)、TCPの改良はOS任せで進化が遅い、という不満があった。QUICはUDP(ポート443)の上に、再送・輻輳制御・ストリーム多重化・TLS 1.3相当の暗号化をひとまとめに実装し、これらを解決する。ストリームごとに独立して再送するため、1ストリームの喪失が他を止めない。QUICの上でHTTPを動かしたものがHTTP/3で、詳しくは第7章で触れる。

合格の視点「UDPは信頼性がないから劣る」ではなく、「信頼性の作り込みをどの層が担うかの役割分担」と捉えると、DNS・RTP・QUICが一本の線でつながる。試験でも「UDPを使う理由を述べよ」(→再送より即時性優先/1対多が必要/軽量な1往復で足りる、のどれか)という形で問われる。

5-10netstatで見るコネクションの状態

ここまでの知識は、サーバでnetstat(またはss)コマンドを打つと現実の姿で見える。表示される「状態」の列は、TCPコネクションが確立・切断シーケンスのどこにいるかを示す状態機械そのものだ。障害解析の午後問題では、この状態名から「いま何が起きているか」を逆算させる。

状態意味これが大量にあるときの読み筋
LISTENサーバが接続を待ち受け中正常(サービス起動の確認に使う)
SYN_SENTSYNを送り応答待ち(クライアント側)相手に届いていない/FWで落ちている疑い
SYN_RECEIVEDSYNを受けSYN/ACKを返し、最後のACK待ち大量ならSYN Flood攻撃の兆候(第8章)
ESTABLISHED確立済み。データ転送中正常。数はサーバの同時接続数
FIN_WAIT_1/2自分からFINを送り、相手の応答/FIN待ち相手アプリの終了処理が遅い
CLOSE_WAIT相手のFINを受けACK済み。自分のアプリのclose待ち大量なら自ホストのアプリがcloseを呼んでいない(アプリの実装・リソースリーク疑い)
TIME_WAITアクティブクローズ完了後の2MSL待機短命接続を大量にさばいた側では正常

表5-2: 主なTCP状態と障害解析での読み方。「CLOSE_WAITの山=自分のアプリの問題」「TIME_WAITの山=たいてい正常」の対比が実務でも試験でも役に立つ。

とくにCLOSE_WAITは理解の試金石だ。この状態は「相手はもう切りたいと言った(FIN受領済み)のに、こちらのアプリケーションがまだクローズしていない」ことを意味する。ネットワークの問題ではなく自ホスト上のアプリケーションの問題を指す状態であり、「netstatの結果からどこに原因があるか推定せよ」という設問の答えの型になる。トランスポート層の状態表示が、アプリケーション層のバグを指し示す——層をまたいで原因を切り分ける感覚は、NW午後試験が一貫して求める力だ。

Q1. ウェルノウンポートとプロトコルの組合せとして、適切なものはどれか。
  • ア SMTP — TCP 110
  • イ DNS — UDP 53
  • ウ SNMP — TCP 123
  • エ HTTPS — TCP 80

正解: イ — DNSの名前解決は主にUDP 53(ゾーン転送等ではTCP 53も使う)。アの110はPOP3で、SMTPは25。ウの123はNTPで、しかもSNMPもNTPもUDP。エの80はHTTPで、HTTPSは443。誤答肢はどれも「番号は実在するが対応が1つずれている」形なので、表5-1の対応を正確に。

Q2. 3ウェイハンドシェイクで、クライアントがSYN(シーケンス番号x)を送った直後にサーバが返すセグメントとして、適切なものはどれか。
  • ア SYNフラグのみを設定し、ACK番号は含めない
  • イ SYNとACKフラグを設定し、ACK番号にxを設定する
  • ウ SYNとACKフラグを設定し、ACK番号にx+1を設定する
  • エ ACKフラグのみを設定し、ACK番号にx+1を設定する

正解: ウ — 2番目のセグメントはSYN/ACKで、SYNは1シーケンス番号を消費するためACK番号は「次に欲しい」x+1になる。イは+1を忘れたパターンで最も紛らわしい——ACK番号は「受け取った番号」ではなく「次に欲しい番号」だ。エはハンドシェイク3番目(クライアント側)の形。

Q3. TCPのTIME_WAIT状態に関する記述として、適切なものはどれか。
  • ア 先にFINを送った側が、最後のACKの喪失に備えて2MSLの間コネクション情報を保持する状態である
  • イ 相手からFINを受け取った側が、自アプリケーションのクローズ処理を待つ状態である
  • ウ SYNを送信して相手からの応答を待っている状態である
  • エ TIME_WAITが大量にあるサーバは、アプリケーションの異常を疑う必要がある

正解: ア — TIME_WAITはアクティブクローズ側が2MSL待つ状態。目的は最終ACK喪失時のFIN再送への応答と、旧セグメントの新コネクションへの混入防止。イはCLOSE_WAITの説明で、本問最大の引っかけ。エは逆で、短命接続を大量に自分から切る側にTIME_WAITが積み上がるのはおおむね正常な動作だ(異常を疑うべきはCLOSE_WAITの山)。

Q4. TCPの高速再転送(fast retransmit)が発動する契機として、適切なものはどれか。
  • ア 再送タイマ(RTO)が満了したとき
  • イ 同一のACK番号をもつ確認応答を3回重複して受信したとき
  • ウ 受信側からウィンドウサイズ0が通知されたとき
  • エ SACKオプションで受領済み範囲が通知されたとき

正解: イ — 歯抜けの後続データが届くたびに受信側は同じACKを返すので、3回の重複ACK=「その番号のセグメントだけが失われた」強いサイン。タイマを待たず即再送する。アはタイムアウト再送で、高速再転送とは対になる別の契機。ウはフロー制御(送信停止)の話。エのSACKは「何を再送すべきか」を絞り込む補助情報であり、発動契機そのものではない。

Q5. TCPのフロー制御と輻輳制御に関する記述として、適切なものはどれか。
  • ア フロー制御は途中のネットワークの混雑を、輻輳制御は受信側バッファの溢れを防ぐ
  • イ フロー制御のウィンドウ値は受信側が通知し、輻輳ウィンドウは送信側が自ら推測して調節する
  • ウ 送信可能なデータ量は、受信側の広告ウィンドウと輻輳ウィンドウの大きいほうで決まる
  • エ 輻輳制御では、ルータが混雑度を数値で送信元に通知することで送信量を調節する

正解: イ — rwndは受信側からの申告、cwndは送信側の自主規制。アは説明が逆。ウは「大きいほう」が誤りで、正しくは小さいほう(min)。エのような明示的通知(ECN)も存在はするが、TCPの輻輳制御の基本はパケット喪失を輻輳のサインとみなす暗黙の推測であり、「ルータが数値で通知」は基本動作の説明として不適切。

Q6. RTTが80ミリ秒の回線で、TCPウィンドウサイズが32kバイト(32×103バイト)のとき、スループットの理論上の最大値はおよそ何Mビット/秒か。
  • ア 0.4Mビット/秒
  • イ 3.2Mビット/秒
  • ウ 25.6Mビット/秒
  • エ 32Mビット/秒

正解: イ — 32×103バイト×8 = 256×103ビットを0.08秒ごとに送れる。256×103 ÷ 0.08 = 3.2×106ビット/秒 = 3.2Mビット/秒。アは×8を忘れて÷1000した誤り、ウはRTTを8ミリ秒と取り違えた(桁ミス)値、エはバイト→ビット変換だけしてRTTで割り忘れた値。単位変換を先に片付けてから割るのが安全だ。

Q7. MTUが1500バイトのイーサネットにおける、TCPのMSSの標準的な値はどれか。
  • ア 1440バイト
  • イ 1460バイト
  • ウ 1480バイト
  • エ 1500バイト

正解: イ — MSS = MTU − IPヘッダ20 − TCPヘッダ20 = 1460。ウ(1480)はTCPヘッダを引き忘れた値(=IPペイロード長)、アはヘッダを30バイトずつなどと誤った値、エはMTUそのもの。「MSSはTCPデータ部だけ」という定義から引き算を組み立てられれば迷わない。

Q8. ファイアウォールでICMPを全て遮断したところ、あるサーバとのTCP接続は確立するが、サイズの大きいデータ転送だけが途中で止まるようになった。原因として最も適切なものはどれか。
  • ア ICMPの遮断によりARPが機能せず、MACアドレスを解決できなくなったから
  • イ 経路MTU探索で使うICMPエラーが届かず、経路上のMTUを超えるパケットが破棄され続けるから
  • ウ ICMPの遮断によりTCPのACKが返らなくなり、再送が繰り返されるから
  • エ pingによる死活監視が失敗し、ルータが経路をダウンと判定したから

正解: イ — PMTUDはICMPタイプ3コード4のエラーに依存する。遮断されると送信元は経路上の小さいMTUを学習できず、DF付きの大きなパケットが黙って捨てられ続ける。小さいパケット(ハンドシェイク)は通るので「接続はできるが大きな転送で固まる」という症状になる。アは誤り——ARPはICMPと無関係のL2プロトコル。ウも誤りで、ACKはTCPセグメントでありICMPではない。エのpingが経路判定に使われる事実はない。対策(タイプ3コード4の許可、MSSクランプ)までセットで覚えよう。

Q9. トランスポート層にUDPを採用しているプロトコルの組合せとして、適切なものはどれか。
  • ア DNS(名前解決)、NTP、SNMP
  • イ SMTP、POP3、IMAP4
  • ウ HTTP/1.1、FTP、Telnet
  • エ SSH、HTTPS(HTTP/2まで)、FTP

正解: ア — 小さな問合せと応答が1往復で済むDNS・NTP・SNMPはUDPが基本(DNSはゾーン転送等でTCPも使う)。イはメール系でいずれもTCP——確実に届くことが最優先の用途だ。ウ・エもすべてTCP上のプロトコル。「1往復で済む軽い通信・リアルタイム系・1対多」がUDP、「確実な転送が必要な対話・ファイル転送」がTCP、という用途の軸で判別できる。

Q10. あるサーバでnetstatを実行したところ、特定のアプリケーションのコネクションがCLOSE_WAIT状態で大量に残っていた。この状況の説明として、最も適切なものはどれか。
  • ア 通信相手がFINに応答しないため、切断が完了できずにいる
  • イ 相手からのFINは受信済みだが、自サーバのアプリケーションがコネクションのクローズ処理を行っていない
  • ウ 短時間に大量の接続を自分から切断した直後の正常な状態である
  • エ SYN Flood攻撃を受けており、ハンドシェイクが完了しない接続が滞留している

正解: イ — CLOSE_WAITは「相手のFINにACKは返したが、自分のアプリがまだcloseしていない」状態。大量に滞留していれば自ホスト上のアプリケーションの実装(クローズ漏れ)を疑う。アは相手待ちの状態(FIN_WAIT系)の説明で方向が逆。ウはTIME_WAITの説明。エの滞留はSYN_RECEIVEDに現れる。状態名から「どちら側の・どの層の問題か」を切り分けるのがこの問題の狙いだ。
